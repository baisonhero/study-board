# サプライチェーン攻撃

ソフトウェアの開発・配布・依存関係の「信頼の連鎖」を悪用する攻撃手法の総称。直接ターゲットを攻撃するのではなく、ターゲットが信頼しているサードパーティ（ライブラリ、ツール、インフラ）を侵害することで間接的に攻撃する。

## なぜ重要か

- 現代のソフトウェアは大量のOSS依存関係を持つ（平均的なNode.jsプロジェクトで数百〜数千パッケージ）
- 1つの上流パッケージの侵害が、下流の無数のプロジェクトに波及する
- 開発者は依存パッケージを「信頼済み」として扱うため、検知が遅れやすい
- CI/CDパイプラインの自動化が攻撃の伝播を加速する

## 攻撃の分類

### 1. 依存関係の悪用

| 手法 | 説明 | 事例 |
|------|------|------|
| **Dependency Confusion** | 内部パッケージと同名のパッケージを公開リポジトリに登録し、ビルド時に優先的にインストールさせる | 2021年 Alex Birsan の研究で Apple / Microsoft / PayPal 等に影響 |
| **Typosquatting** | 人気パッケージに似た名前（`lodahs` など）で悪意あるパッケージを公開 | npmやPyPIで継続的に発生 |
| **メンテナアカウントの乗っ取り** | OSSメンテナのアカウントを侵害し、既存パッケージに悪意あるコードを注入 | 2018年 event-stream事件（npmパッケージ、暗号通貨窃取コードが注入） |
| **悪意あるPR/コミット** | OSSリポジトリに正当に見えるPRで悪意あるコードを紛れ込ませる | 2021年 Linux カーネルへのUMN大学による意図的な脆弱性注入実験 |

### 2. ビルド・配布インフラの侵害

| 手法 | 説明 | 事例 |
|------|------|------|
| **ビルドシステムの侵害** | CI/CDパイプラインやビルドサーバーを侵害し、正規のビルド成果物に悪意あるコードを注入 | 2020年 SolarWinds事件（ビルドプロセスにバックドア注入、米政府機関を含む18,000組織に影響） |
| **パッケージレジストリの侵害** | npm, PyPI, Maven Central 等のレジストリ自体を攻撃 | 2024年 xz-utils事件 |
| **コード署名鍵の窃取** | 正規の署名鍵を盗み、悪意あるコードに正規の署名を付与 | 2022年 NVIDIA のコード署名証明書流出 |

### 3. 開発ツールの侵害

- IDE拡張機能への悪意あるコード注入
- 開発者向けツール（linter、formatter等）の侵害
- コンテナイメージの改ざん

## 主要なインシデント

### SolarWinds（2020年）
- SolarWindsのOrion製品のビルドプロセスにSUNBURSTバックドアが注入された
- 約18,000の組織がアップデート経由で感染、米政府機関・Fortune 500企業を含む
- 攻撃者はビルドサーバーに長期間潜伏し、正規のアップデートにバックドアを埋め込んだ
- **教訓**: ビルドパイプラインのセキュリティ、署名検証、ゼロトラストの重要性

### Log4Shell（2021年） → [[Log4Shell]]
- Apache Log4j（Javaの広範に使われるロギングライブラリ）にRCE脆弱性（CVE-2021-44228）
- 直接的なサプライチェーン攻撃ではないが、単一の依存ライブラリの脆弱性がいかに広範な影響を及ぼすかを示した
- **教訓**: SBOMの重要性（自分のソフトウェアが何に依存しているか把握する必要性）

### xz-utils（2024年）
- Linux の圧縮ユーティリティ xz に、数年かけて信頼を獲得したメンテナがバックドアを仕込んだ
- sshd に影響するバックドアで、発見が遅れればLinuxサーバーの大規模な侵害につながるところだった
- Andres Freund が偶然パフォーマンス異常に気づいて発見
- **教訓**: 長期的なソーシャルエンジニアリング、OSSメンテナへの信頼の限界、レビューの重要性

### event-stream（2018年）
- 人気npmパッケージ（週200万DL）のメンテナンスを引き継いだ人物が悪意あるコードを注入
- 特定の暗号通貨ウォレット（Copay）の秘密鍵を窃取するコードが含まれていた
- **教訓**: メンテナの引き継ぎ時の審査、依存関係の定期的な監査

## 防御策

### SBOM（Software Bill of Materials）
ソフトウェアの「部品表」。自分のアプリケーションがどのライブラリのどのバージョンに依存しているかを一覧化する。

- **[[SPDX]]**: Linux Foundationが策定したSBOMの標準フォーマット（ISO/IEC 5962:2021）
- **[[CycloneDX]]**: OWASPが策定したSBOMフォーマット。セキュリティユースケースに強い
- SBOMがあることで、新たな脆弱性が発見された際に「自分のプロダクトに影響があるか」を即座に判断できる

### 依存関係の監視とスキャン
- **[[Dependabot]]**: GitHubの依存関係自動更新・脆弱性アラート機能
- **[[Trivy]]**: コンテナイメージ・ファイルシステム・Gitリポジトリの脆弱性スキャナー
- `npm audit` / `pip audit` などパッケージマネージャー組み込みの監査機能
- Renovate: Dependabotの代替となる依存関係更新ツール

### ビルド・配布の整合性検証
- **SLSA（Supply-chain Levels for Software Artifacts）**: Googleが提唱するサプライチェーンセキュリティのフレームワーク。ビルドの再現性・来歴（provenance）を段階的に保証
  - Level 1: ビルドプロセスの文書化
  - Level 2: ホストされたビルドサービスの使用
  - Level 3: ビルドプラットフォームのハードニング
- **Sigstore**: OSSのコード署名・検証の仕組み。署名の敷居を下げる
- **ロックファイルの活用**: `package-lock.json`, `yarn.lock` などで依存バージョンを固定し、整合性ハッシュを検証

### 開発プロセスでの対策
- 依存パッケージの追加時にレビューを行う（`npm install` を無意識に実行しない）
- 最小権限の原則をCI/CDパイプラインに適用する
- プライベートレジストリを使用し、Dependency Confusionを防ぐ
- 依存パッケージの数を最小限に抑える
- `--ignore-scripts` フラグでインストール時スクリプトの自動実行を防ぐ

## 自分のプロジェクトでやるべきこと

Next.js + Supabase + Vercel スタックでの実践:

1. **Dependabot有効化** → [[Dependabot]]（CLAUDE.mdのDevSecOpsルールにも記載済み）
2. **`npm audit`の定期実行** をCI/CDに組み込む
3. **ロックファイルをコミット** して依存バージョンを固定
4. **SBOMの生成** を検討（`@cyclonedx/cdxgen` や `spdx-sbom-generator` など）
5. **Trivyでコンテナスキャン** を導入（Dockerを使う場合）→ [[Trivy]]
6. **GitHub Security Advisories** のウォッチ
