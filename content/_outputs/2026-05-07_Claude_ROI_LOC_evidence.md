## 4. 言語スコープ（重要：単一言語ではなく言語混合の全プロジェクト）

### 4.1 結論

表 A1-2-1 のスコープは **「主開発言語 1 が識別できる新規開発プロジェクト全体」**。特定言語のみ抽出ではなく、言語混合プロジェクトを含む全数集計である。「言語1が不明」のレコードのみ除外している。

### 4.2 一次ソース根拠

**(a) 同一サンプルの SLOC 規模×工数を扱う A1.1.2 の本文記述（p.82）**

> 「ここでは、新規開発ですべての言語混在のプロジェクトを対象に、SLOC 規模と工数の関係を図 A1-1-2 に示す。」

— 同じ層別条件で動く隣接節が「すべての言語混在」と明示しており、A1.2.1 / 表 A1-2-1 も同様に解釈できる。

**(b) 抽出取り決め（p.175 A4.2.1）**

> 「規模がコード行数（SLOC）の場合は、｢主開発言語 1｣の名称が明確であること。」

— 「言語が判別できないレコードを除外」する条件であり、「特定言語のみ」を意味しない。

**(c) 多言語プロジェクトの取り扱い（p.217 A5.2.7）**

> 「開発言語が複数言語の場合、言語別に上位 5 言語について次の項目。プロジェクト内で使用言語の規模の多いものから順に記載する。」

— 主開発言語 1 はプロジェクト内で SLOC が最大の言語を指す。複数言語プロジェクトが当然に含まれる。

## 5. 分布の解釈と外れ値

### 5.1 形状

平均 1.27 KSLOC/人月 vs 中央値 0.60 KSLOC/人月の 2 倍超の乖離、SD=6.66 という巨大な分散、最大 114.28 KSLOC/人月の存在から、分布は **対数正規型に近い右裾の長い分布**。「±n σ」基準は意味が薄い。

### 5.2 数学的な μ+3σ（参考値）

`1.27 + 3 × 6.66 = 21.25 KSLOC/人月 ≈ 21,000 LOC/人月`

ただし対数正規分布で σ ベースの上限を引くのは統計的に不適切。参考値としてのみ示す。

### 5.3 Tukey 外れ値判定（推奨）

`IQR = P75 − P25 = 1.00 − 0.36 = 0.64`
`上限フェンス = P75 + 1.5 × IQR = 1.00 + 0.96 = 1.96 KSLOC/人月 ≈ 2,000 LOC/人月`

**「実用的な上限閾値 ≒ 2,000 LOC/人月」**。これを超える観測値は外れ値として扱う。

### 5.4 最大値 114.28 KSLOC/人月の解釈

中央値の約 190 倍、平均の約 90 倍。原データレベルでは存在するが、自動生成コード・コピーペースト多用・極端な低密度プロジェクトなど、人間の月産生産能力とは別物の観測。ROI 議論のベースラインとしては除外すべき。

## 6. ROI レポートで使うべき数字

| 採用シナリオ | 値 | 用途 |
|---|---|---|
| 保守的（中央値） | **600 LOC/人月** | 最も安全な業界平均ベースライン |
| 楽観的（P75） | **1,000 LOC/人月** | 高生産プロジェクト想定 |
| 平均（参考） | 1,270 LOC/人月 | 平均が外れ値で歪むことの注記用 |
| 上限フェンス（Tukey） | 約 2,000 LOC/人月 | これを超える主張は要警戒 |
| 最大値 | **使用しない** | 114K LOC/人月は外れ値 |

**推奨レンジ：600〜1,000 LOC/人月**。中央値〜P75 のレンジで提示することで、IPA 原典の N=307 全数の分布の中で「半数〜上位 1/4」の範囲を示せる。

## 7. ROI レポートに貼れる「前提」段落（ドラフト）

> **業界ベースラインの設定根拠**
> 業界平均の月産 LOC は、IPA「ソフトウェア開発分析データ集2022」全体版 p.84 表 A1-2-1「SLOC 規模別 SLOC 生産性（新規開発）」を一次ソースとして採用する。同表は N=307 の新規開発プロジェクトを対象に、実効 SLOC（コメント・空行除外）を開発 5 工程の実績工数で除した値を集計したもので、1 人月＝160 人時で換算している。中央値は 0.60 KSLOC/160 人時、P75 は 1.00 KSLOC/160 人時、すなわち **600 〜 1,000 LOC/人月** が業界中央〜上位 25% のレンジに相当する。対象は「主開発言語 1 が明確な」プロジェクトであり、言語混合プロジェクトを含む全数集計（うち Java が約 4 割、COBOL が約 16%、Visual Basic.NET / C# / C が後続）である。平均値は 1.27 KSLOC/160 人時だが、最大値 114.28 KSLOC/160 人時という外れ値の影響で右に裾を引いた対数正規型分布のため、本レポートでは中央値〜P75 のレンジを基準値として採用する。

## 8. 議論の経緯

1. **当初** — 業界平均月産 LOC を 800〜1,200 で議論していた（出典不明、推定ベース）。
2. **IPA 一次ソースで再確認** — 表 A1-2-1 の中央値 0.60 KSLOC/160人時 = 600 LOC/人月、P75 = 1.00 = 1,000 LOC/人月を発見。一次ソース直引用が議論として圧倒的に強いため、レンジを **600〜1,000** に着地。
3. **「3σは？」議論** — 平均 1.27 / SD 6.66 から μ+3σ = 21,000 LOC/人月という値が出たが、平均と中央値が 2 倍超乖離していることから分布が対数正規型と判明。±σ ベースの上限判定は不適切と確認。
4. **代替判定の採用** — Tukey の IQR 法で上限フェンス約 2,000 LOC/人月を導出。これを「外れ値判定の実用ライン」として採用。
5. **言語スコープの最終確認**（本ドキュメント Step 1） — 表 A1-2-1 は単一言語抽出ではなく、主開発言語 1 が識別できる全プロジェクト（言語混合含む）を対象。Java 約 4 割を中心に COBOL / VB.NET / C# / C が含まれる構成と確認。

## 9. 代替シナリオ

**(A) 600〜1,000 LOC/人月（推奨）**
新規開発の中央値〜P75 を採用。Java 中心・言語混合の業界一般像にフィットする。

**(B) 改良開発主体なら 480 LOC/人月（IPA 表 A1-2-2 改良開発の中央値）**
改修・保守・拡張プロジェクトが主体の組織には、表 A1-2-2「SLOC 規模別 SLOC 生産性（改良開発）」p.87 の中央値 0.48 KSLOC/160 人時 = **480 LOC/人月** を使う。N=522 とサンプルが多く信頼性が高い。

| 表 A1-2-2 改良開発 | KSLOC/160人時 |
|---|---|
| N | 522 |
| 中央 | 0.48 |
| P75 | 0.78 |

**(C) 自社プロジェクト実測でローカル補正**
