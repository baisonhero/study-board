# シークレット管理

> [!summary]
> APIキー・トークン・DB接続情報・暗号鍵などの「秘密」をどこにどう保管し、どう配布し、どう破棄するか。`.env` をリポジトリに入れない、Gitに混入したら即rotation、漏洩を検知する仕組みを作る。

## シークレットの分類

- **静的シークレット** — 長期間使う鍵（DBパスワード、APIキー）
- **動的シークレット** — 使うたびに発行される短命トークン（Vault dynamic credentials、AWS STSの一時クレデンシャル）
- **証明書 / 暗号鍵** — TLS証明書、署名鍵、アプリ間mTLS

可能な限り **動的シークレット** に寄せるのが現代のベストプラクティス。漏洩しても寿命が短ければ被害が限定される。

## ローカル開発の運用

### `.env` の扱い

- `.env` は**絶対にGitにコミットしない**（`.gitignore`で除外、コミット履歴も含めて）
- `.env.example` をリポジトリに置いてキー名と用途だけ共有（値はダミー）
- 異なる環境用に `.env.local` `.env.development` を分ける（Next.js規約）

### 開発者ローカルでの保管

- **1Password CLI**（`op`）— Vaultから動的に取得して環境変数化
- **Doppler** — クラウドベース、複数開発者で共有しやすい
- **direnv** — 単純なファイルベース、プロジェクトディレクトリに入った時だけロード
- **AWS SSO + temporary credentials** — AWSへのアクセスは長期キーを置かない

ターミナル履歴に直書きしない（CLAUDE.mdのDevSecOpsルールに記載済み）。

## CI/CDでのシークレット

- **GitHub Actions Secrets** / **GitHub Environments**
- **Vercel Environment Variables**（Production / Preview / Development で分ける）
- **AWS Secrets Manager** + IAM Role
- 生のキーをworkflowログに出さない（`::add-mask::` でマスク、もしくはランタイム取得）

> [!warning] PR からシークレットが漏れる経路
> Pull Request から実行される workflow に `secrets.*` を渡すと、悪意あるPRでシークレットが奪取される。`pull_request_target` 使用時は要注意。

## 本番ランタイムでのシークレット

| 場所 | 推奨 |
|---|---|
| AWS環境 | **Secrets Manager** または **SSM Parameter Store**（後者は安価） |
| Kubernetes | **External Secrets Operator** 経由でクラウドVault連携、生のSecretオブジェクト直書きは避ける |
| Vercel/Netlify | プラットフォームの Environment Variables 機能 |
| Supabase | Vault機能（Postgres pgsodium ベース） |
| 自己ホスト | **HashiCorp Vault** |

### 取得タイミング

- **起動時に取得してメモリに保持**（一般的）
- **リクエストごとに取得**（短命シークレットや高頻度ローテーション時）
- **キャッシュ + TTL**（バランス重視）

## 漏洩防止：シークレットスキャン

### Pre-commit でブロック

- **gitleaks** — リポジトリ全体スキャン、pre-commit hook化可能
- **TruffleHog** — エントロピーベース検出も可能、過去履歴スキャン
- **detect-secrets**（Yelp製）— ベースラインで運用しやすい

### サーバ側

- **GitHub Secret Scanning**（パブリックリポジトリで自動有効、private は Advanced Security）
- **GitGuardian**（SaaS）

## 漏洩した時の対応（Rotation）

検知から rotation までの流れ：

1. **直ちにキーを無効化（revoke）** — まずここから
2. アクセスログを精査して被害範囲を特定
3. 新しいキーを発行
4. すべての使用箇所を更新
5. インシデントノート（[[インシデントレスポンス]]）に記録

> [!tip] Rotationは「やりやすい設計」が大事
> シークレットを差し替えるのに何時間もダウンタイムが必要な設計だと、漏洩時に rotation を躊躇する。**通常運用でもrotation を回す前提**で組むと、緊急時にスムーズ。

## 暗号鍵管理（KMS）

- **AWS KMS** / **GCP KMS** / **Azure Key Vault** — 鍵を取り出さずに暗号化・復号操作を依頼するモデル
- **Envelope Encryption** — データ鍵をKMSで暗号化、暗号化済みデータ鍵をデータと一緒に保存
- **HSM** — 物理的な耐タンパー鍵保管（規制要件で必要なとき）

## 個人プロジェクトでの最低ライン

1. `.env` を `.gitignore` に確実に入れる
2. gitleaks を pre-commit hook に
3. Vercel/SupabaseのENVに本番値を入れて、コードに埋め込まない
4. AWSは IAM Role + 一時クレデンシャル、Access Key 長期発行を避ける
5. 万一漏らしたら **5分以内に revoke** できる手順を頭に入れておく
