# インシデントレスポンス

> [!summary]
> セキュリティインシデント発生時の **検知 → 対応 → 復旧 → 事後分析** の流れ。「侵害が起きた前提」で何を残すか・誰がどう動くかを事前に決めておく活動。CLAUDE.md の `#incident` タグ運用と直結する。

## 4フェーズ（NIST SP 800-61 ベース）

```
Preparation ─┬─> Detection & Analysis ─> Containment, Eradication, Recovery ─> Post-Incident Activity
             └────────────────────────── 全フェーズで継続的改善 ──────────────────────────┘
```

| フェーズ | 主な活動 |
|---|---|
| **Preparation** | ランブック整備、ツール準備、訓練、連絡網 |
| **Detection & Analysis** | 検知、トリアージ、影響範囲特定 |
| **Containment** | 拡散防止（隔離、アクセス遮断） |
| **Eradication** | 原因除去（マルウェア駆除、脆弱性修正） |
| **Recovery** | 通常運用復帰、再発しないことの確認 |
| **Post-Incident** | ポストモーテム、再発防止策、ランブック更新 |

## 個人開発でも最低限揃えておくこと

### Preparation（一番大事）

- **連絡フロー**：誰に通知する？（Vercelサポート、Supabaseサポート、ユーザー、ブログ告知）
- **ランブック**：「APIキー漏れた」「DBに不正アクセス」「DDoS来た」等のシナリオ別手順書
- **ログの保全**：すぐ消えてしまう CloudWatch Logs / Vercel Logs を別ストレージに退避する手順
- **アクセスキーのrotation手順**：5分以内に revoke できる状態
- **連絡先のオフライン保管**：本番が落ちててもアクセス情報を見られる場所に

### Detection

- アラート設計：何が「異常」なのかを事前に定義
- 観測手段：[[OpenTelemetry]] / Vercel Analytics / Supabase Logs
- 異常パターン例：
  - 認証失敗の急増
  - 普段アクセスのないIPからの管理APIアクセス
  - 課金額の異常増加
  - エラー率の急上昇

### Containment（封じ込め）

優先順位：**拡大を止める > 原因究明**。証拠保全のためにすぐに環境を破壊しない判断と、被害拡大を止めるためにすぐ遮断する判断、両方を場面で切り替える。

選択肢：
- アクセスキーの即時 revoke
- 関連IAMロールの一時無効化
- ドメインのDNSを切り替え（メンテナンスページへ）
- 該当機能のフラグでOFF

### Eradication & Recovery

- 脆弱性パッチ適用
- 改ざんされたデータの巻き戻し（バックアップから）
- 認証情報の総rotation
- 「再発しないこと」をテストで保証してからオープン

### Post-Incident（ポストモーテム）

**Blameless Postmortem** が原則。「誰がミスしたか」ではなく「なぜそうなる仕組みだったか」に焦点。フォーマット例：

```markdown
## 概要
## 影響範囲
## タイムライン
## 根本原因（5 Whys）
## 検知の遅れ要因
## 再発防止策（Action Items）
## 学び
```

CLAUDE.mdのルールに従い、Vaultに `#incident` タグでノートを残す。

## ロギングの設計（IR起点で考える）

「**事故が起きた後でも追える**」ように設計する：

- ユーザー識別子（PIIではないハッシュ化したID）
- リクエストID（[[OpenTelemetry]] のtraceId）
- アクション種別（read / write / delete / admin-op）
- 結果（success / fail）
- 失敗理由（**ただしクレデンシャルや個人情報を含めない**）
- 認証情報の使用記録（誰のキーで誰が叩いた）

[[セキュリティロギング設計]] で詳述。

## クラウド事業者対応

- AWS — IAM Access Analyzer、CloudTrail（**全リージョンで有効**にしておく）、GuardDuty
- Vercel — Logs、Audit Log（Pro以上）
- Supabase — Database logs、Auth logs

クラウド事業者側のサポート連絡経路を**事前に手元にメモ**しておく。事故発生中にサポートポータル探してる時間は無い。

## 規制対応の観点

- GDPR — 個人データ侵害は **72時間以内**に監督機関に通知義務（[[コンプライアンスと法規制]]）
- 日本の個人情報保護法 — 個人情報保護委員会への報告と本人通知の義務
- PCI DSS — カード情報の侵害時の通知要件
- SOC 2 — インシデント管理プロセスが監査対象

## 訓練（Tabletop Exercise）

「もし○○が起きたらどうする？」をテキスト上で時系列に追うシミュレーション。15〜30分で1ケース、月1〜四半期1で回す。

ランブックの抜けを見つける効果が大きい。

## OWASP Top 10との対応
