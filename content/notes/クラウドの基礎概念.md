# クラウドの基礎概念

> [!summary]
> IaaS / PaaS / SaaS のサービスモデル、リージョン / AZ / エッジ、責任共有モデル、価格モデル、可用性。AWS全冠保持なら大半は知ってる前提だが、**用語のベンダー横断的な意味**と他クラウドとの対応を整理する。

## サービスモデル（XaaS）

### 自前 → IaaS → PaaS → SaaS の階層

```
                  自前      IaaS      PaaS     SaaS
アプリ            自        自        自        ベンダ
ランタイム        自        自        ベンダ    ベンダ
OS                自        自        ベンダ    ベンダ
仮想化            自        ベンダ    ベンダ    ベンダ
サーバHW          自        ベンダ    ベンダ    ベンダ
ネットワーク      自        ベンダ    ベンダ    ベンダ
データセンター    自        ベンダ    ベンダ    ベンダ
```

### 各モデルの特徴と例

| モデル | 自分で管理 | ベンダ管理 | 例 |
|---|---|---|---|
| **IaaS** | OS/MW/アプリ/データ | HW/仮想化/ネット | AWS EC2、GCP Compute Engine、Azure VM |
| **PaaS** | アプリ/データ | OS/MW/ランタイム | AWS Elastic Beanstalk、Heroku、Vercel、Cloud Run |
| **CaaS** | コンテナイメージ | コンテナ実行基盤 | ECS、Fargate、GKE、Cloud Run |
| **FaaS** | 関数コード | 実行環境 | AWS Lambda、Cloud Functions、Vercel Functions |
| **SaaS** | （設定のみ） | 全部 | Salesforce、Slack、GitHub、Supabase |

> [!tip] FaaS と CaaS の使い分け
> 短時間・イベント駆動 → FaaS、長時間・カスタム実行環境 → CaaS。Lambda は **15分上限**があるので、長時間処理はECS/Cloud Runへ。

## リージョン / AZ / エッジ

### リージョン（Region）

地理的に独立したデータセンター集合。AWS の `ap-northeast-1`（東京）、`us-east-1`（バージニア北部）等。

選ぶ基準：
- **ユーザーの近さ**（レイテンシ）
- **法規制**（[[コンプライアンスと法規制]]、データレジデンシー）
- **対応サービス**（新サービスは us-east-1 から）
- **コスト**（リージョンで価格差）

### Availability Zone（AZ）

リージョン内の独立したデータセンター（複数）。AWSは多くのリージョンで2〜6個。

- **障害ドメインを分離**：1AZ落ちても他AZで継続
- マルチAZ構成が**可用性の基本**
- AZ間レイテンシは1〜2ms程度

### エッジロケーション

コンテンツ配信用の小さい拠点。世界数百か所に分散。

- **CDN**（CloudFront、Cloudflare、Fastly）
- **エッジコンピューティング**（Lambda@Edge、Cloudflare Workers、Fastly Compute）

## 責任共有モデル（Shared Responsibility Model）

「**クラウド側 vs 顧客側**でセキュリティ責任が分かれる」概念。

### IaaSの場合（EC2 等）

| クラウド側責任 | 顧客側責任 |
|---|---|
| 物理セキュリティ | OS パッチ |
| ハイパーバイザ | アプリケーション |
| ネットワークインフラ | ネットワーク設定（SG、ACL） |
| 仮想化 | データ暗号化（at-rest含む） |
| | IAM 設定、ユーザー管理 |

### PaaS / SaaS では顧客側責任が減る

PaaS（Lambda、Cloud Run 等）は OS とランタイム部分も Cloud 側責任に。SaaS だとアプリケーション本体も。

> [!warning] よくある誤解
> 「クラウド使ってるからセキュリティはクラウド側がやってくれる」 ← **間違い**。データ・設定・IAMは常に顧客側責任。S3バケット公開で漏えいする事故が後を絶たないのはこの誤解が原因の一つ。

## 主要な価格モデル

### EC2 / VM の場合

| モデル | 概要 | 適する場面 |
|---|---|---|
| **On-Demand** | 使った時間分だけ | 不規則・短期 |
| **Reserved Instance / Savings Plans** | 1年/3年確約で割引（最大72%） | 安定運用、予測可能 |
| **Spot** | 余剰キャパを安く（最大90%）、中断あり | バッチ、フォールトトレラント |
| **Dedicated Host** | 物理サーバ占有 | ライセンス・コンプラ要件 |

### サーバレス / 従量

- Lambda：実行時間 + 呼び出し回数
- API Gateway：リクエスト数
- DynamoDB：読み書きユニット or オンデマンド
- S3：ストレージ + リクエスト + 転送

「**ゼロまでスケールダウン**」できるのが魅力。ただし**過剰呼び出しでコスト爆発**するリスクあり、課金アラート必須。

### データ転送料金

クラウドの「**隠れコスト**」：

- インターネットへの**egress** は基本有料
- 同一リージョン内、同一AZ内は通常無料
- リージョン間は有料
- VPC Peering、Transit Gateway も有料

CDNを噛ますとオリジンからのegressが減らせる。

## 可用性指標

### SLA（Service Level Agreement）

クラウドが保証する稼働率：

| 9の数 | 稼働率 | 月間ダウンタイム上限 |
|---|---|---|
| Two 9s | 99% | 7時間20分 |
| Three 9s | 99.9% | 43分 |
| Four 9s | 99.99% | 4分20秒 |
| Five 9s | 99.999% | 26秒 |

EC2 単一インスタンス：99.5%、マルチAZ：99.99% 等、構成で大きく変わる。

### MTTR / MTBF

- **MTTR**（Mean Time To Recovery）— 障害から復旧までの平均時間
- **MTBF**（Mean Time Between Failures）— 障害間隔の平均

## ベンダーロックイン

クラウド固有サービスを使うほど、別クラウドへの移行コストが上がる：

- **完全マネージド系**（DynamoDB、Cosmos DB、Spanner）— ロックイン強い
- **OSS 互換系**（RDS、PostgreSQL系、Redis系）— 移行しやすい
- **Kubernetes** — マネージドK8s（EKS / GKE / AKS）はクラウド間で移行可能性高い

ロックインを避けたいなら：
- データはOSS互換DB（PostgreSQL、Redis）
- ワークロードはコンテナ化
- IaCで定義（[[IaCとTerraform基礎]]）

完全な avoid は非現実的。**重要部分だけ移植性を確保**する戦略が現実的。

## マルチクラウド / ハイブリッドクラウド

| パターン | 特徴 | 例 |
|---|---|---|
| **シングルクラウド** | 1社に集中 | AWSのみ |
| **マルチクラウド** | 複数クラウドを並行使用 | AWS + GCP（用途別） |
| **ハイブリッドクラウド** | クラウド + オンプレ | AWS + 社内DC |
| **マルチリージョン** | 同じクラウド内複数リージョン | AWS東京 + バージニア |

マルチクラウドの動機：
- ベンダーロックイン回避
- リージョン障害対策
- ベスト・オブ・ブリード（GCPはML、AWSはサービス幅、Azureは法人IT）
- 法規制（特定国のデータは特定クラウド）

ただし運用複雑度は跳ね上がる。**安易に手を出さない方がいい**。

## 主要クラウドのざっくり棲み分け

| 観点 | AWS | GCP | Azure | Cloudflare |
|---|---|---|---|---|
| 強み | サービス幅、エコシステム | ML/データ分析、ネット | エンタープライズ、Office連携 | エッジ、無料枠 |
| 弱み | UIが煩雑、価格 | 地域カバー、シェア | クセが強い | コンピュート系は限定的 |
| 価格戦略 | 個別課金 | 比較的シンプル | プラン制 | 無料枠厚い |

詳しくは [[AWSとGCPとAzureの比較]]。
