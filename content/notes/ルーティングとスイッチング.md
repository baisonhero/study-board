# ルーティングとスイッチング

> [!summary]
> パケットが「**どこを通って届くか**」の仕組み。L2スイッチが同じネットワーク内の転送、L3ルーターが異なるネットワーク間の転送を担う。AWSのRoute Tableも本質的に同じ仕組み。

## L2スイッチング（同じネットワーク内）

### MACアドレスとは

- **48ビット**のNIC固有識別子（例：`00:1A:2B:3C:4D:5E`）
- 上位24ビット = ベンダID（OUI）、下位24ビット = ベンダ内通し番号
- グローバル一意（理論上）

### スイッチの動き

L2スイッチは **MACアドレステーブル** を持っていて、「このMACはこのポートに居る」を学習：

1. パケット受信時、送信元MACをポートと紐付けて記憶
2. 宛先MACがテーブルに有る → そのポートだけに転送
3. 無い → 全ポートにフラッディング

### ARP（Address Resolution Protocol）

「このIPを持つホストのMACを教えて」を解決する仕組み（IPv4の場合）：

```
A: who has 192.168.1.10? tell 192.168.1.1     (broadcast)
B: 192.168.1.10 is at 00:1A:2B:3C:4D:5E       (unicast to A)
```

ARPキャッシュは `arp -a` または `ip neigh` で見える。

IPv6では **NDP**（Neighbor Discovery Protocol）が同じ役割。

## L3ルーティング（異なるネットワーク間）

### ルーティングテーブル

OSやルーターが持つ「**この宛先IPはここに送れ**」の表：

```
$ ip route
default via 192.168.1.1 dev eth0      # デフォルトルート
192.168.1.0/24 dev eth0               # 直接接続
10.0.0.0/8 via 192.168.1.254          # 別ネットへの経路
```

優先順位は **最長プレフィックスマッチ**：より具体的（プレフィックス長が長い）なルートが優先。

### デフォルトゲートウェイ

「ルーティングテーブルにマッチしない宛先」を投げる先。家庭・社内なら社内ルーター、AWSのVPCなら**Internet Gateway**や**NAT Gateway**。

### 静的ルーティング vs 動的ルーティング

| 項目 | 静的 | 動的 |
|---|---|---|
| 設定 | 手動 | プロトコルで自動学習 |
| 規模 | 小〜中 | 中〜大 |
| 障害対応 | 手動 | 自動再経路 |
| 例 | AWS Route Table（多くは静的） | OSPF、BGP |

クラウド環境はほぼ静的（マネージドが裏で管理）。BGPは大規模ネットワークやインターネット間で重要。

## ルーティングプロトコル（参考）

### IGP（内部ゲートウェイプロトコル）

組織内で使う：

- **OSPF**（Open Shortest Path First）— リンクステート型、企業内の標準
- **EIGRP** — Cisco独自
- **RIP** — 古い、距離ベクトル

### EGP（外部ゲートウェイプロトコル）

ISP間、AS（自律システム）間：

- **BGP**（Border Gateway Protocol）— 唯一の標準。インターネットの根幹

> [!info] BGPがクラウドでも重要な場面
> AWS Direct Connect、Azure ExpressRoute、専用線・VPN接続でBGPを話す必要が出る。クラウド+オンプレ混在環境で必須知識。

## AWS VPCの「ルーティング」

VPCは仮想ネットワーク。**Route Table** で経路制御：

```
[Subnet A] ──→ [Route Table A]
                ├ 10.0.0.0/16 → local（VPC内）
                ├ 0.0.0.0/0   → Internet Gateway（パブリック）
                └ 10.1.0.0/16 → VPC Peering Connection
```

サブネット単位でRoute Tableを紐づける。**Public Subnetは「IGWへのルートがある」サブネット**、Private Subnetは無い。

### 主要なターゲット

- **Internet Gateway (IGW)** — インターネット双方向
- **NAT Gateway** — プライベートからインターネットへ片方向
- **VPC Endpoint** — AWSサービスへ私設経路（S3、DynamoDB等）
- **Transit Gateway** — 複数VPCのハブ
- **VPC Peering** — 2つのVPC間
- **Virtual Private Gateway** — オンプレへVPN

## VLAN（Virtual LAN）

物理スイッチを仮想的に分割。L2レベルで隔離。

- **VLAN ID** は1〜4094（12ビット）
- **タグ付け（dot1q）**：フレームに4バイト付加
- **アクセスポート**（タグなし、特定VLAN専用）vs **トランクポート**（複数VLANをタグ付きで運ぶ）

クラウドではあまり意識しないが、**AWS VPCがマルチテナントで隔離を実現する仕組み**は VLAN の仮想拡張版（VXLAN）に近い概念。

## STP（Spanning Tree Protocol）

L2でループが発生すると**ブロードキャストストーム**で破滅する。STPがループを自動検出してリンクをブロック：

- ルートブリッジを選定
- ループになるパスを止める
- 障害時に自動切り替え

クラウドでは見えないが、オンプレ運用で必須概念。

## トラブルシューティングコマンド

```bash
# ルーティングテーブル確認
ip route                          # Linux
ip -6 route                       # IPv6

# 特定の宛先への経路を表示
ip route get 8.8.8.8

# ARPテーブル
ip neigh                          # Linux
arp -a                            # クロスプラットフォーム

# 経路追跡
traceroute 8.8.8.8                # UDP/ICMP
mtr 8.8.8.8                       # 連続実行版、便利

# AWSの経路確認はマネジメントコンソール or CLI
aws ec2 describe-route-tables
```

## ありがちなトラブル

- **「Pingは通るのに通信できない」** → L4以降の問題（FW、ポート、アプリ）
- **「経路がない」** → Route Tableに該当ルート追加忘れ
- **「Public Subnetなのにインターネットに出れない」** → IGWアタッチ忘れ、ルート未追加、Public IPなし
- **「VPC Peering通らない」** → 双方のRoute Tableに相手レンジを書く必要、Security Group設定
- **「重複アドレス」** → VPN/Peering先と CIDR重複
