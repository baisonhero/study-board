# 認証と認可の違い

> [!summary]
> **認証 (Authentication, AuthN)** は「**あなたは誰か**」を確認するプロセスで、**認可 (Authorization, AuthZ)** は「**そのあなたに何をする権利があるか**」を判定するプロセス。混同されやすいが、設計上は明確に分けて扱う必要がある。混同すると [[A01 Broken Access Control]] (認可漏れ) と [[A07 Identification and Authentication Failures]] (認証強度不足) の両方を同時に踏みやすい。詳細は [[認証と認可]] 参照。

## 一行で言うと

- **認証 (AuthN)** = WHO are you?（パスポート提示）
- **認可 (AuthZ)** = WHAT can you do?（VIPラウンジに入る権利）

## プロセスの順序

```
[ログイン] → AuthN（誰だっけ） → セッション/トークン発行 → AuthZ（この人は何できる） → 業務実行
```

AuthN は通常1セッションに1回、AuthZ は **リクエストごと** に判定される。

## 具体例

| 動作 | カテゴリ |
|---|---|
| パスワード入力 | AuthN |
| TOTP入力 | AuthN（多要素） |
| 生体認証 | AuthN |
| [[Passkey]] 提示 | AuthN |
| 「admin だけ /admin にアクセス可」 | AuthZ |
| 「自分の投稿だけ削除できる」 | AuthZ |
| [[Supabase RLS]] | AuthZ |
| OWASP A07 | AuthN の問題 |
| OWASP A01 | AuthZ の問題 |

## 実装パターン

### 認証 (AuthN)

- セッションクッキー方式（古典）
- JWT (JSON Web Token) ベアラ
- OIDC / SAML 経由のSSO
- Passkey / WebAuthn
- mTLS (クライアント証明書)

### 認可 (AuthZ)

- **RBAC (Role-Based)**: ロール単位で権限を持つ。管理画面の標準
- **ABAC (Attribute-Based)**: 属性条件で動的に判定 (`user.dept == resource.dept`)
- **ReBAC (Relationship-Based)**: Google Zanzibar 流。「ユーザーが文書のオーナー」の関係グラフで判定
- **PBAC (Policy-Based)**: OPA / Rego のような汎用ポリシー言語

## よくある混同事例

- 「ログインさえできれば何でもできる」: 認証が認可を兼ねている → 多くのインシデント原因
- 「JWTにロールを書いてあるから安全」: トークン署名は AuthN だが、ロール検証 (AuthZ) はリクエストごとにサーバーが行わないと意味がない
- 「フロントエンドで /admin を隠した」: 認可はサーバー側で行う必要がある (フロントは UX のためだけ)

## 設計上のチェックリスト

- [ ] 認証 (AuthN) 失敗時の挙動: タイミング攻撃に注意した一定時間レスポンス
- [ ] 認可 (AuthZ) は**サーバーで毎リクエスト**判定
- [ ] 認証情報が JWT に乗るなら、`exp` / `iss` / `aud` を必ず検証
- [ ] 認可ポリシーは **デフォルト拒否** の設計
- [ ] 監査ログには **誰が** **何に** **どの権利で** **何をしたか** を残す ([[A09 Security Logging and Monitoring Failures]])
