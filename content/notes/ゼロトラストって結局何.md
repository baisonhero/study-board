# ゼロトラストって結局何

> [!summary]
> **ゼロトラスト (Zero Trust)** はネットワーク内部にいるからといって信頼しない、というセキュリティモデル。`Never trust, always verify` がスローガン。境界型 (Perimeter-based) セキュリティ ＝「社内ネットワーク内なら安全」というモデルが、リモートワーク・クラウド・SaaSの世界で破綻したため、置き換えとして提唱された。Forrester (2010)、NIST SP 800-207 (2020)、Google BeyondCorp (2014) が代表的な定式化。詳細は [[ゼロトラストとネットワーク基礎]] 参照。

## なぜ必要か

従来モデル:

```
[Internet] - [FW] -------- [社内LAN: 何でもアクセス可]
                            ↑
                       VPNで入れば信頼
```

問題:

- VPNが破られる/認証情報が漏れると **全社が即座に侵害**
- 内部の悪意ある社員・侵入後のラテラルムーブ防げず
- リモートワーク / クラウドSaaS / モバイルデバイス時代に**「内部」の定義が崩壊**

## ゼロトラストの基本原則

1. **すべてのアクセスを毎回検証**: ネットワーク位置で信頼しない、毎リクエストで認証・認可
2. **最小権限**: ジョブ機能に必要な分だけ
3. **Assume Breach**: 既に侵入されている前提で設計（横展開を防ぐ）
4. **デバイス信頼**: 業務端末のセキュリティ状態 (パッチ、EDR) を継続検査
5. **暗号化**: 内部通信もすべて暗号化（mTLS, [[WireGuard]], [[TLS]]）
6. **可観測性**: すべてのアクセスを記録、異常を検知

## 主要な構成要素

| カテゴリ | 実装例 |
|---|---|
| ID/認証 | Okta / Azure AD / Auth0、[[Passkey]] / MFA |
| デバイス | Jamf / Intune / Kandji + EDR (CrowdStrike) |
| ネットワーク | [[Tailscale]] / Cloudflare Access / Zscaler |
| アプリ | SSO 統合、API ゲートウェイ、[[Supabase RLS]] |
| データ | [[CSPM]]、[[CNAPP]]、暗号化、機密データ分類 |
| 可観測性 | SIEM、SOAR、[[Observability]] |

## Google BeyondCorp の影響

Google が 2009年のオーロラ攻撃を契機に開始。**「VPNを廃止し、すべてをインターネット越しに、ただし強固な認証で守る」** を10年かけて構築。社員はホテルのWi-Fiから直接社内アプリに繋ぎ、IDプロバイダ + デバイス証明書 + ポリシーで認可される。今では Cloudflare Access / Tailscale / Zscaler が「BeyondCorp相当を SaaS で提供」している。

## 「ゼロトラスト製品」のマーケティング過熱

ベンダーは何でも「ゼロトラスト対応」と謳う。実態の見分け方：

- **VPNを置き換えると謳いつつ、ネットワーク中心**: 本物のZTNAなら個別アプリ単位の認可
- **デバイス信頼を含まない**: 半分のゼロトラスト
- **継続検証 (Continuous Verification)** の有無
- **DataやAPIへの認可レイヤーまでカバーするか**

## 実装は段階的

完全なゼロトラストへの移行は数年がかり。現実的な段階：

1. **MFA全社展開**: 最も投資対効果が高い
2. **SSO / IdP統合**: アプリへの認証経路を1つに
3. **デバイス信頼**: MDM + 証明書ベース認証
4. **VPNからZTNAへ**: アプリ単位のアクセス制御 ([[Tailscale]] / Cloudflare Access)
5. **マイクロセグメンテーション**: 内部通信もmTLS
6. **データ・API認可の継続検証**
