# IPアドレスとサブネット

> [!summary]
> IPv4/IPv6、CIDR記法、サブネット計算、プライベート/パブリック。AWS VPCを設計する時に必ず使う基礎。「/24は何台入る？」「10.0.0.0/16を/24に分割すると？」が即答できるレベルが目標。

## IPv4の基本

- **32ビット**を**8ビット×4オクテット**で表現：`192.168.1.10`
- 各オクテットは0〜255（2^8 = 256通り）
- 約43億通り（実際にはほぼ枯渇、IPv6への移行が進む）

## IPv6の基本

- **128ビット**を**16ビット×8グループ**を16進数で：`2001:0db8:85a3:0000:0000:8a2e:0370:7334`
- 連続する `0000` グループは `::` で省略可能（1回だけ）
- 約340澗通り（実質的に無限）
- IPv4と並走（デュアルスタック）

## CIDR記法

サブネットマスクをスラッシュで表記：

- `192.168.1.0/24` = サブネットマスク 255.255.255.0
- 数字 = ネットワーク部のビット数
- 残りがホスト部

### 主要なCIDRと収容ホスト数

| CIDR | サブネットマスク | ホスト数（理論値） | 利用可能 |
|---|---|---|---|
| /24 | 255.255.255.0 | 256 | 254 |
| /25 | 255.255.255.128 | 128 | 126 |
| /26 | 255.255.255.192 | 64 | 62 |
| /27 | 255.255.255.224 | 32 | 30 |
| /28 | 255.255.255.240 | 16 | 14 |
| /29 | 255.255.255.248 | 8 | 6 |
| /30 | 255.255.255.252 | 4 | 2 |
| /23 | 255.255.254.0 | 512 | 510 |
| /22 | 255.255.252.0 | 1024 | 1022 |
| /16 | 255.255.0.0 | 65536 | 65534 |
| /8 | 255.0.0.0 | 16M | 16M-2 |

「利用可能」がホスト数より2少ない理由：**ネットワークアドレス**（最初）と**ブロードキャストアドレス**（最後）は使えない。

> [!warning] AWS VPCはさらに5つ予約
> AWSのサブネットでは、最初の4つと最後の1つを予約しているので、`/24` で利用可能なのは **251個**（254 - 3）。VPC設計時に小さいサブネットを切ると思った以上に枯渇する。

## サブネット計算の練習

### 例: 10.0.0.0/16 を /24 に分割

- /16のホスト部 = 16ビット
- /24のホスト部 = 8ビット
- サブネット部 = 16 - 8 = 8ビット → 256個のサブネット

結果：`10.0.0.0/24`、`10.0.1.0/24`、`10.0.2.0/24`、... 、`10.0.255.0/24`

### 例: 192.168.1.0/24 を /28 に分割

- /28 のホスト部 = 4ビット → 16アドレス/サブネット
- /24 を /28 に → 16個のサブネット

結果：`192.168.1.0/28`、`192.168.1.16/28`、`192.168.1.32/28`、... 、`192.168.1.240/28`

各サブネットの「使える範囲」（AWS以外）：
- `192.168.1.0/28` → ホスト `192.168.1.1` 〜 `192.168.1.14`
- ネットワーク `192.168.1.0`、ブロードキャスト `192.168.1.15`

## プライベートIPアドレス（RFC 1918）

組織内・宅内で自由に使える範囲。インターネット直接到達不可。

| 範囲 | CIDR | 用途 |
|---|---|---|
| 10.0.0.0 〜 10.255.255.255 | `10.0.0.0/8` | 大企業、データセンター |
| 172.16.0.0 〜 172.31.255.255 | `172.16.0.0/12` | 中規模 |
| 192.168.0.0 〜 192.168.255.255 | `192.168.0.0/16` | 家庭、SOHO |

AWS VPCも基本的にこのレンジから選ぶ。

## 特殊なIPアドレス

| アドレス | 意味 |
|---|---|
| `0.0.0.0/0` | デフォルトルート、全IPv4 |
| `127.0.0.1/8`、`::1` | ループバック（自分自身） |
| `169.254.0.0/16` | リンクローカル（DHCP失敗時） |
| `169.254.169.254` | クラウドのインスタンスメタデータエンドポイント（AWS、GCP、Azure共通） |
| `224.0.0.0/4` | マルチキャスト |
| `255.255.255.255` | 制限ブロードキャスト |

> [!warning] メタデータエンドポイントとSSRF
> `169.254.169.254` はAWSでもクラウド全般でメタデータ取得用。ここに**外部からSSRFで到達されると認証情報が抜かれる**ので、AWSは IMDSv2（トークン必須）を強制すべき。[[Web脆弱性の実装防御]] のSSRFセクション参照。

## NAT（Network Address Translation）

プライベートIP ↔ パブリックIPの変換。

- **NAT**：1対1の変換
- **PAT / NAPT**（Port Address Translation）：1つのパブリックIPを複数プライベートIPで共有（家庭ルーターはこれ）
- **Source NAT（SNAT）**：送信元IPを変換（外向き）
- **Destination NAT（DNAT）**：宛先IPを変換（ポートフォワーディング）

AWS：
- **NAT Gateway** — マネージド、SNAT
- **Elastic IP** — 静的パブリックIPの予約
- **VPC Endpoint** — NATを介さずAWSサービスと通信

## VPC設計の実例

### ありがちなVPC設計（/16 を分割）

```
VPC: 10.0.0.0/16

Public Subnet:
  10.0.1.0/24 (AZ-a)
  10.0.2.0/24 (AZ-c)

Private Subnet (App):
  10.0.10.0/24 (AZ-a)
  10.0.11.0/24 (AZ-c)

Private Subnet (DB):
  10.0.20.0/24 (AZ-a)
  10.0.21.0/24 (AZ-c)
```

各AZにPublic / Private / Private(DB) の3層。/24 単位で256アドレス、AWS予約引いて約250利用可能。

### 重複に注意

VPCピアリングや VPN接続するなら、**他社や本社のレンジと重複しない**ようにする。`10.0.0.0/8` を全部使ってる組織と接続できなくなる罠。

## 確認コマンド

```bash
# IPアドレスとマスク確認
ip addr show              # Linux モダン
ifconfig                  # Linux 古い、Mac
ipconfig                  # Windows

# ルーティングテーブル
ip route                  # Linux モダン
route -n                  # Linux 古い
netstat -rn               # 多くのOS

# ARPテーブル（IP→MAC）
ip neigh                  # Linux モダン
arp -a                    # 多くのOS
```
