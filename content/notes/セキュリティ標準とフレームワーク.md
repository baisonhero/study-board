# セキュリティ標準とフレームワーク

> [!summary]
> 自分の組織の成熟度を測る・外部に説明する・脆弱性の優先順位を決めるための「共通言語」。NIST CSF、ISO 27001、CIS Benchmarks、CVSS、OWASP SAMM 等。各々の役割と使いどころを把握する。

## なぜフレームワークが必要か

セキュリティは**比較**と**説明**ができないと進まない。

- 自社の状態を客観的に測る
- 経営や顧客に「現状」と「投資効果」を説明する
- 異なる組織や監査人と同じ語彙で話す
- ベンダー選定で**比較可能な軸**を持つ

「うちはちゃんとやってる」では誰も納得しない世界。

## NIST Cybersecurity Framework（CSF）

### 概要

- 米国立標準技術研究所（NIST）が公開
- 元は重要インフラ向けだが、**業種を問わず広く使える**
- v2.0（2024年公開）で大幅刷新

### 6つの機能（v2.0）

| 機能 | 内容 |
|---|---|
| **Govern**（v2新設）| ガバナンス、リスク管理、ロール定義 |
| **Identify** | 資産・リスク・脅威の特定 |
| **Protect** | アクセス制御、データ保護、訓練 |
| **Detect** | 検知の仕組み |
| **Respond** | インシデント対応 |
| **Recover** | 復旧計画と改善 |

各機能の下にカテゴリ・サブカテゴリがあり、**チェックリスト的に自社実装を埋めていける**。

### 個人開発者でも使える形

軽量に使うなら：

1. 6機能それぞれで「自分が何をやってるか」を1〜2行で書く
2. ギャップを発見
3. 来期の優先課題に

これだけでも考え漏れが激減する。

## ISO/IEC 27001（ISMS）

### 概要

- 国際標準の **情報セキュリティマネジメントシステム**
- 認証取得には外部審査が必要
- 日本で「セキュリティ認証取ってます」と言うとほぼコレ
- ISO 27002 が具体的な管理策（114個 → 2022改訂で93個）

### SOC 2 との比較

| 項目 | ISO 27001 | SOC 2 |
|---|---|---|
| 出自 | ISO（国際） | AICPA（米国会計） |
| 性質 | プロセスの正式認証 | 監査報告書（Type 1/2） |
| 取得期間 | 6か月〜1年以上 | Type 2 で6か月〜1年 |
| 認知 | 日本・欧州で強い | 米国B2B SaaSで標準 |
| 費用 | 年間数百万〜 | 同程度 |

事業展開する地域・顧客で選ぶ。両方取る企業も多い。

## CIS Benchmarks

### 概要

- Center for Internet Security が公開
- 個別技術製品（OS、クラウド、DB等）の **具体的な設定推奨値**
- AWS、Linux、Windows、Kubernetes、Docker、ブラウザ等の Benchmark あり
- **Level 1**（基本）/ **Level 2**（高セキュリティ）

### 使い方

- AWS Security Hub の `CIS AWS Foundations Benchmark` で自動チェック
- Linux サーバには **CIS-CAT** や Ansible role で設定
- Kubernetes は **kube-bench** で監査

「**何を設定すべきか具体的に教えてくれる**」のが価値。NIST CSFが「何をすべきか」、CISが「どう設定するか」。

## CVSS（Common Vulnerability Scoring System）

詳細は [[CVSS]] を参照。要点だけ：

### 概要

- 脆弱性の深刻度を **0〜10** で標準化したスコア
- 現行は **v3.1**、**v4.0**（2023年発表）が新標準
- CVE（脆弱性ID）と組み合わせて報告される

### スコアの構成（v3.1）

3つのメトリクス群：

- **Base** — 脆弱性自体の特性（攻撃ベクトル、複雑さ、必要権限、影響）
- **Temporal** — 時間で変化する要素（exploit 公開、パッチ有無）
- **Environmental** — 環境固有の要素（重要資産か等）

| スコア範囲 | レベル |
|---|---|
| 0.1〜3.9 | Low |
| 4.0〜6.9 | Medium |
| 7.0〜8.9 | High |
| 9.0〜10.0 | Critical |

### v4.0 の改善

- "Threat" メトリクスでExploitabilityをより細かく
- "Supplemental" メトリクスで自動化可能性等を追加

### 使うときの注意

CVSSは**深刻度の指標であってリスクの指標ではない**。Critical でも自社環境で使ってない機能なら影響なし、Medium でも認証バイパスなら最優先、ということがよくある。

## OWASP SAMM

[[OWASP]] のソフトウェア保証成熟度モデル。**組織のAppSec成熟度**を測る。

### 5つのビジネス機能 × 3つのプラクティス

| 機能 | プラクティス例 |
|---|---|
| **Governance** | 戦略 / ポリシー / 教育 |
| **Design** | 脅威評価 / セキュリティ要件 / アーキテクチャ |
| **Implementation** | セキュアビルド / セキュアデプロイ / 欠陥管理 |
| **Verification** | アーキテクチャ評価 / 要件検証 / セキュリティテスト |
| **Operations** | インシデント / 環境管理 / 運用管理 |

各々を **Maturity Level 0〜3** で自己評価。改善ロードマップを引ける。

## CWE（Common Weakness Enumeration）

- 「脆弱性の**型**」のカタログ（CVEは個別事象、CWEは型）
- 例：CWE-79（XSS）、CWE-89（SQLi）、CWE-22（Path Traversal）
- [[OWASP Top 10]] の各カテゴリは複数のCWEを束ねたもの
- **CWE Top 25** が「最も危険な脆弱性タイプ TOP 25」として有名

## STRIDE / MITRE ATT&CK

[[脅威モデリング]] のノートで詳述。STRIDEは**設計時の脅威分類**、ATT&CKは**実観測された攻撃手法のカタログ**。

## 個人プロジェクトの現実的な選び方

| 段階 | 使うフレームワーク |
|---|---|
| 個人開発 | NIST CSFを軽量に + CIS Benchmark を該当製品にだけ |
| 初期事業化 | + OWASP SAMM Level 1 を目指す |
| BtoB SaaS | + SOC 2 Type 1 → Type 2 |
| 規制業界 | + ISO 27001、業界別（PCI DSS、HIPAA等） |
| エンタープライズ | + 全部 |

## 学習リソース

- NIST CSF v2.0 公式（無料）
- CIS Benchmarks（無料、登録必要）
- OWASP SAMM v2.0
- 「The Cybersecurity Body of Knowledge (CyBOK)」（無料、英語、学術寄り）
