# MOC Ideas

> [!summary]
> プロダクトアイデアの **パイプライン**。「思いついた」「裏取りした」「検証中」「却下／棚上げ」を1枚の地図として管理する。CLAUDE.md の方針「作ってから売り先を探す、はしない」を実装する場所。

## なぜこのMOCがあるか

個人開発のアイデアは「面白そう」と「売れる」が違う。ここに思いついた順で全部入れて、課題発見 → 市場検証 → 仕様化 → 開発 のステージごとに状態を持たせ、 **検証されないアイデアが勝手にコード化されない** ようにする。

## ステージ管理

各アイデアノートの frontmatter に `stage:` を付けて状態管理：

| stage | 説明 | 判定基準 |
|---|---|---|
| `seed` | 思いつき。1行でもOK | とにかく書く |
| `research` | 課題の裏取り中 | 当事者ヒアリング3件以上 |
| `validating` | MVP/LPで需要検証中 | 課金意思のある相手の存在 |
| `building` | 開発着手 | 仕様書あり、目標KPI設定済 |
| `live` | リリース済 | 初回ユーザ獲得済 |
| `parked` | 一旦棚上げ | 棚上げ理由を必ず書く |
| `killed` | 却下 | 却下理由を必ず書く |

`#decision` タグで stage 遷移時の意思決定を残す。

## 検証フロー

1. **seed → research**: 当事者（ユーザになり得る人）に少なくとも3件、課題の存在をヒアリング
2. **research → validating**: ペイン・支払意思・現行代替手段が確認できたらMVP/LP
3. **validating → building**: 課金（または明確な利用合意）が取れたら着手
4. **building → live**: ローンチ後30日でアクティベーション率と継続率を判定

## アイデア収集の入り口

- 日常で感じた不便（`_daily/` の「気づき」セクションから昇格）
- ユーザインタビューで挙がった課題（[[MOC Comms]]）
- 公式・OSS の議論で気づいた未充足ニーズ（[[MOC Reading]]）
- 既存プロダクトの隣接領域（自分の専門性で勝てる場所）

## 競合・差別化

- 競合が存在 = 市場が存在のシグナル（無いほうが危険）
- 差別化軸（コスト / UX / 法規対応 / 統合性 / コミュニティ / 速度）から最低1〜2軸で勝てる戦略を持つ
- [[MOC Business]] のマネタイズ・価格設計と必ず接続させる
