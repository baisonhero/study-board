# サイドカーパターン — 補助コンテナで横断機能を切り出す

> [!summary]
> **サイドカー（sidecar）＝ メインアプリと同じ実行ユニット（ECS タスク / K8s Pod）の中に、別コンテナとして相乗りする補助役**。バイクのサイドカー（本体に並走する側車）から命名。アプリ本体は業務ロジックだけに集中させて、ログ収集／メトリクス送信／プロキシ／シークレット注入などの横断機能を別コンテナに切り出す設計。本ノートは定義・仕組み（共有するもの／しないもの）・代表例（CloudWatch agent / OTel Collector / Envoy / Fluent Bit）・代替案との比較（in-process / ホストエージェント）・ECS と K8s での実装をまとめる。[[ECS と Lambda の観測性設計]] の「CloudWatch agent サイドカー化」が腹落ちするのが目的。

関連トピック: [[ECS Task]] / [[Kubernetes Pod]] / [[CloudWatch Agent]] / [[OpenTelemetry Collector]] / [[Service Mesh]] / [[Envoy]] / [[Fluent Bit]]

## 1. ひとことで言うと

**サイドカー ＝ メインアプリのコンテナの隣で一緒に動く、補助役のコンテナ**。同じタスク／ Pod の中に同居して、横断的な機能（ログ・メトリクス・プロキシ・シークレット注入など）を担う。

バイクの **サイドカー**（本体に並走する側車）がそのまま比喩。本体の走行性能は変えず、機能を増やす役割。

```
┌─────────── ECS タスク / K8s Pod ───────────┐
│                                              │
│  ┌──────────┐    localhost    ┌──────────┐ │
│  │  アプリ   │ ──────────────> │   補助   │ │
│  │ (Next.js) │   :4317 など    │ (agent)  │ │
│  └──────────┘                  └──────────┘ │
│        共有: ネットワーク / volume           │
│        同時に起動・同時に終了                │
└──────────────────────────────────────────────┘
```

## 2. 何を共有して、何が独立か

| 共有するもの | 独立しているもの |
|---|---|
| **同じ実行ユニット**（ECS タスク／K8s Pod） | **コンテナイメージ**（別 image、別 OS layer） |
| **ライフサイクル**（一緒に起動・終了する） | **プロセス**（独立した PID 空間） |
| **ネットワーク**（同一 ENI、`localhost` で通信できる） | **ファイルシステム**（マウントしない限り見えない） |
| **マウントすれば volume**（明示的に共有可能） | **再起動ポリシー**（コンテナ単位で設定可能） |
| **同じインスタンス／ホスト** | **環境変数**（コンテナごとに別） |

→ 一番強力なのは **`localhost` 通信が成立する**点。アプリは外部のことを意識せず、隣のコンテナと喋るだけで横断機能が使える。

## 3. 代表例

### 3.1 CloudWatch agent サイドカー（[[Application Signals]] のとき）

```
[Next.js コンテナ] ──OTLP→ localhost:4317 ──> [CloudWatch agent コンテナ]
                                                       │
                                                       ▼
                                                  CloudWatch / X-Ray
```

- アプリは OTel SDK で `http://localhost:4317` に spans / metrics を送るだけ
- CloudWatch agent サイドカーが受け取って AWS バックエンドへ転送
- **アプリは「CloudWatch」のことを一切知らなくていい**
- [[ECS と Lambda の観測性設計]] §9.1 で出てくるのがこれ

### 3.2 OpenTelemetry Collector サイドカー

```
[アプリ] ──OTLP→ localhost:4317 ──> [otel-collector サイドカー] ──> 各種バックエンド
```

CW agent と同じ思想で、バックエンドが Datadog / SigNoz / Grafana など何でも切り替えられる中立形態。

### 3.3 Envoy / Linkerd-proxy（[[Service Mesh]]）

```
[アプリ] ──HTTP→ localhost:15001 ──> [Envoy サイドカー] ──mTLS→ 他サービス
```

- すべての出入りトラフィックを Envoy が代行
- mTLS・リトライ・回路ブレーカ・トラフィック分割を**アプリに書かずに済む**
- Istio / Linkerd / AWS App Mesh の中身がこれ

### 3.4 Fluent Bit / Fluentd（ログ集約）

```
[アプリ] ──stdout / file→ 共有 volume ──> [Fluent Bit サイドカー] ──> S3 / OpenSearch
```

ログを集めて整形・転送する。awslogs ドライバの代替として使われる。

### 3.5 その他

- **Vault Agent**: シークレットを取得して共有 volume に書く
- **DB プロキシ系**: コネクションプールやマイグレーション補助
- **Init container**（サイドカーの兄弟）: 起動前に一度だけ走る初期化用コンテナ

## 4. なぜサイドカーにするのか — 代替との比較

| 方式 | 配置 | 長所 | 短所 |
|---|---|---|---|
| **In-process（ライブラリで吸収）** | アプリのコード内 | 1 コンテナで完結、シンプル | アプリに横断ロジックが混ざる／言語ごとに実装／更新がアプリビルドに連動 |
| **サイドカー** | 同一タスク／Pod 内 | アプリは業務だけに集中／多言語 OK／独立に更新／`localhost` で高速 | コンテナ数が増える／リソース割当の考慮が要る |
| **ホストエージェント（DaemonSet / EC2 単位）** | ホストに 1 つ | リソース効率が良い／全タスクで共有 | Fargate では使えない／タスク粒度の設定が難しい |
| **完全別サービス** | 別タスク／別 VPC | 共有しやすい・拡張容易 | ネットワーク 1 hop 増／可用性管理が増える |

**サイドカーが選ばれる主な理由**

- **言語非依存**: アプリが何で書かれていても、サイドカーは標準プロトコル（OTLP / HTTP / syslog）で受け取る
- **アプリと一緒に動く**: 同時起動・同時終了なので、ライフサイクル不整合が起きない
- **`localhost` が速い**: 同一ネットワーク名前空間なので、ネットワーク経由でもオーバーヘッドが極小
- **Fargate で動く**: ホスト権限が要らない（DaemonSet 不可な環境でも動かせる）

## 5. ECS と K8s での実装

### 5.1 ECS の場合

Task Definition の `containerDefinitions` に複数コンテナを並べる。

```json
{
  "family": "my-app",
  "containerDefinitions": [
    {
      "name": "app",
      "image": "myapp:latest",
      "essential": true,
      "portMappings": [{ "containerPort": 3000 }],
      "environment": [
        { "name": "OTEL_EXPORTER_OTLP_ENDPOINT", "value": "http://localhost:4317" }
      ]
    },
    {
      "name": "cwagent",
      "image": "amazon/cloudwatch-agent:latest",
      "essential": false,
      "portMappings": [{ "containerPort": 4317 }]
    }
  ]
}
```

- 同じタスク → 同じ ENI → `localhost:4317` で繋がる
- `essential: true` のコンテナが落ちるとタスク全体が停止する
- ログ・volume は `mountPoints` で共有可

### 5.2 Kubernetes の場合

Pod の `spec.containers` に複数並べる。

```yaml
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: app
      image: myapp:latest
      env:
        - name: OTEL_EXPORTER_OTLP_ENDPOINT
          value: "http://localhost:4317"
    - name: otel-collector
      image: otel/opentelemetry-collector:latest
      ports:
        - containerPort: 4317
```

- 同じ Pod → 同じ network namespace → `localhost` 通信
- Pod の lifecycle 単位で起動・終了
- ConfigMap / Secret / Volume も両方からマウント可

## 6. ハマりやすい点

- **リソース上限**: サイドカーも CPU / メモリを使う。タスク全体のサイジングで忘れがち。CW agent なら 128 MB / 0.1 vCPU 程度を見ておく
- **起動順**: アプリより前にサイドカーが起動完了している必要があるケース（OTel 送信先がまだ無いと初期スパンを落とす）。ECS の `dependsOn` ／ K8s の `startupProbe` で順序を制御する
- **`essential` の扱い**（ECS）: CW agent を `essential: false` にしておかないと、サイドカーの一時的不調でアプリも巻き込まれる
- **ログの混在**: 両コンテナのログがどちらも CloudWatch Logs に流れる。Log Group を分けるか stream prefix で区別
- **DaemonSet との混同**: ホストレベルのエージェント（DaemonSet）とサイドカー（Pod 内）は別物。**Fargate では DaemonSet が使えないのでサイドカー一択**
- **コンテナ間通信は `localhost`、Pod 外は別**: サイドカー → アプリは `localhost`、サイドカー → 外部サービスは通常のネットワーク経路（VPC ルート・SG・NACL）に乗る

## 7. まとめ

- サイドカーは **「アプリ本体と同じ実行ユニット内に同居する補助コンテナ」**
- ライフサイクル・ネットワークを共有し、`localhost` で安全・高速にやり取りする
- ログ・メトリクス・プロキシ・シークレットなどの **横断機能をアプリ外に切り出す**ための定石
- CW agent / OTel Collector / Envoy / Fluent Bit がよく使われる
- ECS なら同じタスクに 2 つ目以降の `containerDefinitions` を足す、K8s なら同じ Pod に複数 `containers`

[[ECS と Lambda の観測性設計]] §9.1 の「CloudWatch agent をサイドカーコンテナ化」とは、まさに上の **§3.1 のパターン**を ECS タスクに組み込むこと。アプリは `localhost:4317` に OTLP を投げるだけで、Application Signals まで届く。
