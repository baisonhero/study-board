# A04 Insecure Design

> [!summary]
> [[OWASP Top 10]] 2021年版で **新設**。実装ミスではなく **「設計時点で詰んでいる」** ケースを別カテゴリ化したもの。「正しく実装してもなお安全にならない」設計欠陥が対象。脅威モデリング、セキュアバイデザイン、リファレンスアーキテクチャ、悪用ケース（abuse case）の検討で防ぐ。コードを書き始める前のフェーズが勝負。

## どういう脆弱性か

「実装の精度」ではなく「設計の妥当性」の問題。よくあるパターン：

- **認証フローの欠陥** — パスワードリセットが「秘密の質問」だけで突破できる
- **レート制限の欠如** — 設計時にDoS／自動化を想定していない
- **ビジネスロジックの抜け** — 在庫が負数にできる、返金が無限ループできる
- **信頼境界の誤設定** — クライアント側のチェックを信頼している
- **アカウント列挙** — 「メアド存在しません」と「パスワード違います」のメッセージ分岐
- **権限モデルの曖昧さ** — ロール／スコープが後付けで継ぎ足されている
- **データフローの想定漏れ** — 機微情報がログに混入する設計

実装上の対策（パッチ）では根本解決できず、**再設計が必要**。

## 攻撃例

### 秘密の質問だけのパスワードリセット

```
1. 攻撃者が被害者のメアドでリセット要求
2. 「あなたの母親の旧姓は？」→ Facebook プロフィールから判明
3. 即座にパスワード変更可能 → 乗っ取り成功
```

実装は完璧でも設計が破綻している例。

### レート制限なしのクーポン適用

```
POST /api/checkout/apply-coupon
{ "code": "SUMMER20" }
```

1日1回適用の意図でも実装で制限していなければ、botが100枚適用して 100% 割引にできる。

### 在庫が負数になる注文フロー

```javascript
// 危険: トランザクション分離されてない
const stock = await getStock(productId)
if (stock > 0) {
  await createOrder(...)
  await decreaseStock(productId)  // 同時注文で stock が負数に
}
```

ビジネスロジック側で「在庫減算」がアトミックに（DB制約や行ロックで）扱われる設計になってないと、競合で破綻する。

### 機微情報がログに残る設計

「全リクエストをログに出す」設計でクレカ番号やJWTが平文ログに残り、SREが見られる状態になる。

## 防御策

### 1. 脅威モデリングを開発初期に実施

[[脅威モデリング]] のSTRIDE / PASTA / LINDDUN等の手法で、**実装前**に攻撃者視点で穴を探す。

- どんな資産があるか（What are we working on?）
- 何が悪意ある行動か（What can go wrong?）
- 何ができるか（What are we going to do about it?）
- うまく対策できたか（Did we do a good job?）

ツール: [[OWASP Threat Dragon]]、Microsoft Threat Modeling Tool。

### 2. Abuse Case を書く

User Story（正常系）の裏側で、「悪意あるユーザーがどう悪用するか」を Abuse Case として書く。

```
User Story: ユーザーは商品にレビューを投稿できる
Abuse Case: 競合他社が大量のbotで星1レビューを投下する
→ レート制限、登録後N日経過、購入者限定が必要
```

### 3. セキュアバイデザインの原則

- **最小権限** — デフォルトで権限ゼロ、必要な分だけ付与
- **Defense in depth** — 多層防御。1層破られても次がある
- **Fail securely** — エラー時はアクセス拒否側に倒す
- **完全な仲介** — すべてのアクセスを毎回認可チェック
- **オープン設計** — セキュリティを「秘密」に依存しない

### 4. リファレンスアーキテクチャを使う

「ゼロから設計」せず、業界で実績のあるパターンを採用：

- 認証: OAuth 2.0 / OIDC（自前実装しない）
- セッション: フレームワーク標準のCookie実装
- パスワード: KDF（Argon2/bcrypt）
- ロール: RBAC / ABAC のリファレンス実装

### 5. 制限と境界を設計時に決める

- レート制限（IP単位／ユーザー単位／エンドポイント単位）
- ペイロードサイズ上限
- リソース消費上限（CPU時間、メモリ、DB行数）
- 同時セッション数上限

### 6. [[OWASP ASVS]] をチェックリストとして使う

ASVS（Application Security Verification Standard）はセキュリティ要件のリファレンス。Level 1/2/3 の段階で適用範囲を選ぶ。

## 検出手段

- **脅威モデリング** — 設計レビューで早期発見
- **アーキテクチャレビュー** — シニアによる第三者目線
- **[[OWASP ASVS]]** — 要件チェックリスト
- **[[OWASP Cheat Sheet Series]]** — 既知パターンと比較

自動ツールではほぼ検知不能。**人と時間** が必要なカテゴリ。

## 参考事例

- **米国大手金融のパスワードリセット** — 4桁PIN総当たりで突破
- **多くの SaaS の "Forgot Password"** — タイミング差でアカウント存在判定可能
- **EC のクーポン悪用** — 設計時にレート制限を想定してなかった例が多発

## Next.js / Supabase での落とし穴

- **クライアント側だけでフォームバリデーション** — サーバ側で再検証必須
- **Supabase Auth の "確認メール" フロー** — メール送信失敗時の挙動を設計
- **Server Actions の冪等性** — 二重送信時の挙動を設計時に決める
- **コスト爆発リスク（A04 + API4）** — LLM API のような従量課金を呼ぶ設計はレート制限／予算上限を最初から組み込む
