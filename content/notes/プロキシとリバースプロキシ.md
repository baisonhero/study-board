# プロキシとリバースプロキシ

> [!summary]
> プロキシ＝**通信の中継役**。「誰から見たか」で**フォワードプロキシ**（クライアント側）と**リバースプロキシ**（サーバ側）に分かれる。Webサーバ運用、CDN、API Gateway、企業ネットの定番技術。

## フォワードプロキシ vs リバースプロキシ

### フォワードプロキシ（Forward Proxy）

**クライアント側に置く**プロキシ。クライアントが「このプロキシ経由で行ってください」と明示的に指定。

```
[Client] → [Forward Proxy] → [Internet/Server]
```

用途：
- **企業の出口制御**：従業員のWebアクセスを記録・フィルタ
- **匿名化**：送信元IPを隠す（VPN的な使い方）
- **キャッシュ**：社内で同じファイルを何度も取得しない
- **認証経由**：プロキシで認証してから外部アクセス

例：Squid、PAC ファイル設定、`HTTPS_PROXY` 環境変数

### リバースプロキシ（Reverse Proxy）

**サーバ側に置く**プロキシ。クライアントは「サーバ」だと思って通信、実は背後に複数のサーバがある。

```
[Client] → [Reverse Proxy] → [App Server 1]
                          → [App Server 2]
                          → [App Server 3]
```

用途：
- **負荷分散**（[[負荷分散とロードバランサー]] と機能重複）
- **TLS終端**
- **キャッシュ**（CDN的）
- **セキュリティ層**（WAF、DDoS吸収）
- **プロトコル変換**（HTTP/2 ⇄ HTTP/1.1）
- **API Gateway**：認証、レート制限、トランスフォーム

## 主要なプロダクト

### リバースプロキシ／Webサーバ

| 製品 | 特徴 |
|---|---|
| **Nginx** | 軽量・高速、設定がコンパクト。Web運用のデファクト |
| **Apache HTTP Server** | 老舗、機能豊富、`.htaccess` の柔軟性 |
| **HAProxy** | L4/L7両対応、超高性能、DBプロキシでも使われる |
| **Traefik** | コンテナ自動連携、Kubernetes Ingress |
| **Caddy** | HTTPS自動化（Let's Encrypt統合）、設定シンプル |
| **Envoy** | 高機能、Service Meshの土台（Istio等） |

### フォワードプロキシ

| 製品 | 特徴 |
|---|---|
| **Squid** | 老舗、企業の出口プロキシで定番 |
| **Privoxy** | プライバシー重視 |
| **mitmproxy** | デバッグ用、HTTPSも復号できる（要証明書信頼）|

### CDN（実質的にリバースプロキシ）

| サービス | 特徴 |
|---|---|
| **CloudFront** | AWSネイティブ、Lambda@Edge / CloudFront Functions |
| **Cloudflare** | 世界最大規模、Workers、無料枠厚い |
| **Fastly** | VCLでカスタマイズ、リアルタイム性高い |
| **Akamai** | 老舗、エンタープライズ |

### API Gateway

リバースプロキシ + 認証 + レート制限 + ドキュメント：

- **AWS API Gateway**（REST API、HTTP API、WebSocket）
- **Kong**（OSS、エンタープライズ版あり）
- **Tyk**
- **Cloudflare API Gateway**

## Nginx 設定の典型例

リバースプロキシ + TLS終端：

```nginx
server {
  listen 443 ssl http2;
  server_name api.example.com;

  ssl_certificate     /etc/letsencrypt/live/api.example.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

  location / {
    proxy_pass         http://backend:8080;
    proxy_http_version 1.1;
    proxy_set_header   Host $host;
    proxy_set_header   X-Real-IP $remote_addr;
    proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header   X-Forwarded-Proto $scheme;
  }

  location /static/ {
    root /var/www;
    expires 1d;
  }
}

# HTTP → HTTPSへのリダイレクト
server {
  listen 80;
  server_name api.example.com;
  return 301 https://$host$request_uri;
}
```

## X-Forwarded-* ヘッダ

プロキシ経由だと「**実際のクライアントIPが何か**」が分からなくなる。慣習として：

- `X-Forwarded-For` — 元のクライアントIP（多段なら `, ` で連結）
- `X-Forwarded-Proto` — 元のプロトコル（http/https）
- `X-Forwarded-Host` — 元のホスト名
- `X-Real-IP` — 直前のクライアントIP（Nginx 流儀）

新標準として `Forwarded` ヘッダ（RFC 7239）もあるが、X-Forwarded-* がデファクト。

> [!warning] X-Forwarded-For の信頼性
> 攻撃者が直接サーバを叩く時に偽の `X-Forwarded-For` を付けると、ログ上は別IP扱いになる。**信頼できるプロキシ経由でない接続では捨てる**設計が必要。Nginxなら `set_real_ip_from` ディレクティブで信頼するプロキシIPを指定。

## プロキシキャッシュ

リバースプロキシ／CDNが応答をキャッシュして配信元への負荷を減らす：

- **Cache-Control** ヘッダで制御
- `public` / `private`（プロキシキャッシュ可否）
- `max-age=N`（秒）
- `no-store`（一切キャッシュしない）
- `no-cache`（毎回 revalidate）
- `s-maxage=N`（CDN/プロキシ用、ブラウザは max-age 優先）

ETag / Last-Modified による条件付きリクエストで「変更なし」を確認できる。

## プロキシとセキュリティ

### Forward Proxy のリスク

- **HTTPSを復号する場合**は中間者になるので、組織のCA証明書を全端末に入れる必要 → 取り扱いに注意
- ログがプライバシー情報を含む → 保管期間と権限管理

### Reverse Proxy の役割

- **WAF配置層** — WAFはほぼ必ずリバースプロキシ層
- **DDoS吸収** — CDN/CFがエッジで吸収
- **TLS終端＋mTLS** — エッジで認証
- **レート制限**

## サービスメッシュ（参考）

Kubernetes時代の「**サイドカープロキシ**」モデル：

- 各 Pod に Envoy をサイドカー注入
- Pod間通信は全部 Envoy 経由
- mTLS、リトライ、サーキットブレイカ、観測を一元化
- 例：Istio、Linkerd、Consul Connect

「すべての通信にプロキシを挟む」アプローチで、現代のセキュリティ要件を満たす設計の主流の一つ。

## トラブルシューティング

```bash
# プロキシ経由でアクセス
curl -x http://proxy.example.com:8080 https://example.com

# プロキシ環境変数
HTTPS_PROXY=http://proxy:8080 curl https://example.com
NO_PROXY=localhost,.internal curl https://example.com

# Nginx設定テスト
nginx -t

# Nginxアクセスログ
tail -f /var/log/nginx/access.log

# 実際のサーバ証明書確認
echo | openssl s_client -servername example.com -connect example.com:443 2>/dev/null | openssl x509 -noout -subject -issuer -dates
```
