# ネットワークトラブルシューティング

> [!summary]
> 「**繋がらない**」を体系的に切り分けるツールと手順。OSI下層から上層へ順に確認していくのが基本。**ping → traceroute → telnet/nc → tcpdump → アプリログ** の流れを身につける。

## 切り分けの基本フロー

```
L1/L2:  リンクは生きてる？
  │ ip addr → IPアドレス取れてる？ DHCP失敗してない？
  ▼
L3:     ルーティングは正しい？
  │ ping <gateway> → ping <remote IP>
  │ ip route → デフォルトルート見える？
  ▼
L4:     ポートに到達できる？
  │ nc -zv <host> <port>
  │ telnet <host> <port>
  ▼
L7:     アプリは応答してる？
  │ curl -v <url>
  │ サーバ側のログ確認
```

「**下から潰す**」のが鉄則。L7だけ見て直らない時は L3、L4 に降りる。

## ping（L3 ICMP到達性）

```bash
# 基本
ping example.com

# 回数指定
ping -c 4 example.com

# 間隔指定
ping -i 0.5 example.com

# パケットサイズ指定（MTU調査）
ping -s 1500 -M do example.com    # don't fragment
```

注意点：
- ICMP は明示的に **拒否されてる** ことが多い（特にクラウド環境のSG）
- **「ping通らない = 接続不可」とは限らない**
- 「ping通る = 通信OK」とも限らない（ポート別にFW設定があるので）

## traceroute（経路追跡）

```bash
# 基本（ICMP/UDP）
traceroute example.com

# TCP経由（ICMPブロックされてる時）
traceroute -T -p 443 example.com

# 連続表示（より便利）
mtr example.com
mtr -T -P 443 example.com    # TCPで443
```

何のためにやるか：
- どこで止まってるか（自社内なのか、ISP内なのか、相手側なのか）
- ホップ数・RTT
- 経路の途中でルーティングがおかしいか

クラウド内では NLB / IGW がアスタリスク返すことが多く、見えづらい。

## DNS問い合わせ（dig / nslookup）

```bash
# 名前解決確認
dig example.com
dig +short example.com
dig example.com @8.8.8.8         # 特定リゾルバに問い合わせ

# レコード種別指定
dig example.com A
dig example.com MX
dig example.com NS

# 経路追跡
dig +trace example.com

# 逆引き
dig -x 192.0.2.1
```

`/etc/resolv.conf` で使うリゾルバを確認。systemd-resolved 環境では `resolvectl status`。

詳しくは [[DNS実践ガイド]]。

## ポート到達性（telnet / nc）

```bash
# TCP接続できるか
nc -zv example.com 443
telnet example.com 443           # 古典的、Ctrl-] q で抜ける

# UDPは確認難しい
nc -zuv example.com 53           # 戻りパケットでICMP unreachable来ない=送れた可能性

# リッスン側
nc -l 8080                       # 8080で待ち受け
ss -tlnp | grep 8080             # 何が listenしてるか確認
```

**telnetは要らないように見えて手早く便利**。

## ソケット状態（ss / netstat）

```bash
# Listenしてるポート
ss -tlnp                         # TCP listening, ポート, PID
ss -ulnp                         # UDP

# 確立済み接続
ss -tan                          # 全 TCP
ss -tan state established

# 古い OS なら netstat
netstat -tlnp
netstat -an | grep ESTABLISHED
```

`ss` のほうが新しくて速い。`netstat` は非推奨。

## パケットキャプチャ（tcpdump / Wireshark）

```bash
# 特定インターフェースを覗く
sudo tcpdump -i eth0

# 特定ホスト
sudo tcpdump host example.com

# 特定ポート
sudo tcpdump port 443
sudo tcpdump port 53 and udp

# 詳細（パケット中身も）
sudo tcpdump -A -v port 80

# pcap保存（後でWiresharkで開く）
sudo tcpdump -i eth0 -w capture.pcap port 443

# 複合フィルタ
sudo tcpdump 'tcp port 80 and host example.com'
```

GUI で詳細解析するなら **Wireshark**（pcap ファイルを開く）。

> [!tip] tcpdump の鉄則
> 「`-w` でファイルに保存して、後でWiresharkで読む」が捗る。リアルタイムで見たいフィルタが分かってる時だけターミナルで。

## HTTP 詳細確認（curl）

```bash
# 詳細ヘッダ
curl -v https://example.com/

# レスポンスヘッダだけ
curl -I https://example.com/

# 特定IPに向ける（DNSバイパス）
curl --resolve example.com:443:192.0.2.1 https://example.com/

# レスポンス時間の内訳
curl -w "@-" -o /dev/null -s https://example.com/ <<EOF
DNS lookup:    %{time_namelookup}s
TCP connect:   %{time_connect}s
TLS handshake: %{time_appconnect}s
First byte:    %{time_starttransfer}s
Total:         %{time_total}s
EOF

# プロキシ経由
curl -x http://proxy:8080 https://example.com/

# クライアント証明書（mTLS）
curl --cert client.pem --key client.key https://example.com/
```

`curl -v` で実際に送信した／受信したヘッダがすべて見える。

## TLS / 証明書確認

```bash
# サーバ証明書の中身を見る
echo | openssl s_client -servername example.com -connect example.com:443 2>/dev/null | openssl x509 -noout -subject -issuer -dates

# プロトコル・暗号スイートを試す
openssl s_client -connect example.com:443 -tls1_3
openssl s_client -connect example.com:443 -cipher 'ECDHE-RSA-AES256-GCM-SHA384'

# 証明書チェーンを表示
openssl s_client -showcerts -servername example.com -connect example.com:443
```

## 帯域・パフォーマンス

```bash
# 帯域測定
iperf3 -s          # サーバ側
iperf3 -c <server> # クライアント側

# 1パケットあたりのレイテンシ
ping -c 100 example.com

# ファイルダウンロード速度
curl -o /dev/null https://example.com/large.iso  # 速度表示
```

## クラウド固有のツール

### AWS

```bash
# VPC Flow Logs（パケットフロー記録）
aws ec2 describe-flow-logs

# Reachability Analyzer（経路の可達性分析）
aws ec2 create-network-insights-path \
  --source <eni-id> --destination <eni-id> \
  --protocol TCP --destination-port 443

# CloudWatch Logs Insights でログクエリ
fields @timestamp, @message | filter @message like /timeout/
```

### Kubernetes

```bash
# Pod 内から疎通確認
kubectl exec -it <pod> -- curl -v http://svc.namespace.svc.cluster.local

# Network Policy 確認
kubectl get networkpolicies

# DNS 確認
kubectl exec -it <pod> -- nslookup kubernetes.default
```

## 切り分けの考え方（実例）

「APIサーバに接続できない」という症状の時：

1. **DNS解決できる？** → `dig api.example.com`
2. **IPに到達できる？** → `ping <IP>` または `traceroute -T -p 443 <host>`
3. **ポート到達？** → `nc -zv <host> 443`
4. **TLSハンドシェイク成功？** → `openssl s_client -connect <host>:443`
5. **HTTPレスポンスは？** → `curl -v https://<host>/`
6. **応答コード何？** → 5xx ならサーバ側、4xx ならリクエスト側、タイムアウトならネット
7. **サーバログは？** → 該当時刻のアプリログ

各段階で**問題があれば下記の階層に絞って深堀り**。

## ありがちな落とし穴

- **MTU不一致**：VPN経由でtcp/443は通るがpingの大きいサイズで破綻 → MSS clamping
- **DNSキャッシュ古い**：レコード変更直後に古い結果が返る → `sudo systemd-resolve --flush-caches`
- **/etc/hostsで上書きされてる**：DNSは正しいのにホスト指定で迂回されてる
- **IPv6優先で IPv4経路使われない**：dual stackの罠
- **「接続したと思ったらSNI違いで別サーバ」**：`curl --resolve` で意図したIPに行ってない
