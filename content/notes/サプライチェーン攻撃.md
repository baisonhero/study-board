# サプライチェーン攻撃 — 定義・脅威・最新事例・他攻撃との違い

> [!summary]
> **サプライチェーン攻撃**＝ターゲット組織を直接攻撃せず、その組織が信頼している**第三者（OSS 依存パッケージ、ベンダー、CI/CD、SaaS、ハードウェア等）を経由して侵入する攻撃**。1 つの上流を侵害すれば下流の数十〜数万組織に同時波及するため**増幅効果**が極めて高い。本ノートは定義／代表的な類型（OSS / CI/CD / ベンダー / OAuth / ハードウェア）／他攻撃との違い／2024〜2026 年の本当に起きた事例（XZ Utils、Polyfill.io、CrowdStrike、SolarWinds、Axios・TanStack・LiteLLM・Red Hat npm・Vercel OAuth）／脅威の特徴／防衛策の入口、をコンパクトにまとめる。

関連トピック: [[SBOM]] / [[Dependabot]] / [[Trivy]] / [[CI/CD セキュリティ]] / [[OAuth セキュリティ]] / [[ゼロトラストとネットワーク基礎]] / [[A08 Software and Data Integrity Failures]]

## 1. ひとことで言うと

ターゲット A を直接狙わず、**A が信頼して使っている B（ソフトウェア依存、ベンダー、CI/CD、SaaS、ハードウェア）に毒を仕込んで、A を間接的に侵害する**手口。A 側は「正規の更新」「正規のベンダー」だと思って取り込むので、通常のセキュリティ境界を**裏口から無効化**できるのが恐ろしさ。

## 2. 定義と特徴

正式には「**製品・サービスの製造・流通・運用過程に介入して、最終利用者に届く成果物に悪意ある変更を施す攻撃**」。

特徴:

- **信頼の連鎖を悪用**: 標的組織は「正規の更新」と認識して受け入れる
- **増幅効果**: 1 つの上流を侵害すれば下流の数十〜数万組織に同時に波及
- **検知が遅れがち**: 署名済みバイナリ・公式リポジトリ経由なので異常検知が効きにくい
- **責任所在が分散**: 自社・ベンダー・OSS メンテナのどこに非があるか曖昧

## 3. 攻撃の類型

### 3.1 OSS パッケージ・依存ライブラリ

- npm / PyPI / RubyGems などのレジストリに悪意あるバージョンを公開
- メンテナのアカウント乗っ取り、Typosquatting（似た名前で偽パッケージ）、Dependency Confusion（内部名と同名の public 版を出して優先取得させる）
- 代表事例: XZ Utils（2024）、Axios・TanStack・LiteLLM・Red Hat npm（2026）

### 3.2 CI/CD パイプライン

- GitHub Actions / Jenkins / GitLab CI などのビルド過程で改ざん
- ビルド時にシークレットを盗む、成果物に悪意あるコードを混入
- 代表事例: Codecov（2021）、Trivy / Checkmarx（2026 年 3 月）

### 3.3 ベンダー製品・マネージドサービス

- 監視ツール・RMM・SaaS の正規アップデート経路に毒を仕込む
- 代表事例: SolarWinds Orion（2020）、Kaseya VSA（2021）、3CX（2023）、MOVEit（2023）

### 3.4 認証情報・OAuth トークン

- SaaS 間連携の OAuth トークンを乗っ取り、信頼関係を経由して侵入
- 代表事例: **Vercel（2026 年 4 月）** — Context.ai の Google Workspace OAuth トークン経由

### 3.5 ハードウェア

- 製造・輸送過程で物理的に改造、悪意あるチップ・ファームウェアを埋め込む
- 代表事例: ヒズボラ向けポケットベル爆破事件（2024 年 9 月）

## 4. 他の攻撃との違い

| 攻撃タイプ | 標的 | 経路 |
|---|---|---|
| **サプライチェーン攻撃** | 最終利用者組織 | **信頼している第三者経由（間接）** |
| フィッシング | 個別ユーザー | メール・チャットから直接 |
| ランサムウェア | 組織 | 直接侵入後の暗号化（手段の話） |
| ゼロデイ攻撃 | 脆弱性のある組織 | 既知でない脆弱性を直接突く |
| 直接の Web 攻撃 | 公開システム | SQLi / XSS など標的の表玄関 |

**ランサムウェアやデータ流出は「結果」**であって、**サプライチェーン攻撃は「侵入経路の話」**。最近の大規模ランサム被害の多くがサプライチェーン経由で始まる（MOVEit → Clop など）。

## 5. 最近の本当に起きた事例（2024〜2026）

### 5.1 XZ Utils バックドア（2024 年 3 月）

- Linux の圧縮ライブラリ `xz-utils` v5.6.0 / v5.6.1 に**バックドア**が混入
- 「Jia Tan」名義のメンテナが**約 2 年かけて**信頼を積み上げてから悪意あるリリース
- 発覚は Microsoft の Andres Freund 氏が SSH ログイン遅延に気づいたこと
- 影響範囲は Linux ディストリの SSH。間一髪で広範な実害は回避された
- **教訓**: 「**1 人の overworked OSS メンテナ**」が世界中のインフラの critical path に立っているという構造的脆弱性

### 5.2 Polyfill.io ドメイン乗っ取り（2024 年 6 月）

- 多数の Web サイトで使われていた CDN ホスト名 `polyfill.io` が中国系企業に売却 → 悪意ある JS を配信
- 約 10 万サイトに影響、Cloudflare 等が即座にサブスティチュート提供

### 5.3 CrowdStrike Falcon 不具合（2024 年 7 月）

- マルウェアではないが、**サプライチェーン障害**の代表例
- セキュリティ製品の設定ファイル不備で **世界中の約 850 万台の Windows がブルースクリーン**
- 航空・金融・医療が同時停止
- 「**信頼するベンダーの 1 ファイルが世界経済を止めた**」事象として、サプライチェーンリスクを再認識させた

### 5.4 SolarWinds Orion（2020 年、定番として）

- IT 監視ツールのアップデートに Sunburst マルウェアが混入 → 約 18,000 顧客に配布
- 米連邦政府複数機関、FireEye、Microsoft 等が被害
- サプライチェーン攻撃を「**特殊な脅威カテゴリ**」として広く認知させた事件

### 5.5 npm エコシステムへの集中攻撃（2026 年）

2026 年は **npm への集中攻撃が常態化**しており、依存元の検証は実務必須:

- **Axios（2026 年 3 月）**: 主要メンテナの npm アカウント乗っ取り → CI/CD バイパスで改ざんリリース。北朝鮮系の脅威アクターに帰属
- **LiteLLM（2026 年 3 月）**: PyPI で日次 340 万 DL の AI インフラライブラリが改ざん
- **Trivy / Checkmarx（2026 年 3 月）**: Trivy への攻撃を起点に Checkmarx の GitHub リポジトリへ波及
- **TanStack / Mini Shai-Hulud（2026 年 5 月）**: 大規模 OSS への攻撃の一環。**OpenAI 社員端末も巻き込まれ**、macOS の強制アップデートを要した
- **`@redhat-cloud-services` npm（2026 年 6 月）**: Red Hat 公式名前空間配下の **32 パッケージ**が改ざん、コードレビューを丸ごとバイパス

### 5.6 Vercel OAuth サプライチェーン攻撃（2026 年 4 月）

- Context.ai が Lumma Stealer マルウェアに感染（2026 年 2 月頃）
- Context.ai の **Google Workspace OAuth トークン**が盗まれ、これを使って **Vercel の内部システムに侵入**
- 一部 Vercel 顧客プロジェクトの**環境変数が露出**
- 「**SaaS 間の OAuth 連携が新しい攻撃面**」を浮き彫りに

## 6. 脅威の特徴 — なぜ怖いか

1. **信頼を逆手に取る**: SG・WAF・EDR 等の「外を疑う」防御を**裏口から無効化**できる
2. **拡散力が桁違い**: 1 つの上流を侵害すれば数百〜数万組織に同時波及
3. **検知が困難**: 署名済み・公式更新・既知ベンダーなので、異常検知ロジックを通過してしまう
4. **責任分散と賠償の難しさ**: ベンダー・OSS・自社のどこに責任があるか曖昧
5. **「1 人」「1 ファイル」で世界が止まる**: XZ Utils と CrowdStrike が示した、過剰な集中依存

## 7. 防衛策（入口だけ）

完全防御は難しいが、リスクを下げる定石:

- **[[SBOM]]**（Software Bill of Materials）の生成・管理 — 何の依存があるか可視化
- **依存パッケージのバージョン固定** ＋ 自動 PR（[[Dependabot]] / Renovate）
- **CI/CD の Secret 露出最小化** — OIDC で短命トークン、長期キーを置かない
- **ベンダー・SaaS の OAuth スコープ最小化**（必要な権限だけ）
- **ビルド成果物の署名と検証**（Sigstore、SLSA フレームワーク）
- **異常な公開・更新の検知** — Socket.dev、Snyk、GitHub Advanced Security 等
- **インシデント想定の事前訓練** — ベンダー被害時の切り離し手順を Runbook 化（[[インシデントレスポンス]]）
- **第三者リスク管理（TPRM）の仕組み化**

## 8. まとめ

- サプライチェーン攻撃 ＝ **「信頼している第三者を経由して入る」攻撃の総称**
- 形は多様（OSS / CI/CD / ベンダー / OAuth / ハードウェア）
- 2024〜2026 年で頻発（XZ、Polyfill、CrowdStrike、Axios、TanStack、LiteLLM、Red Hat npm、Vercel OAuth…）
- **直接の防御だけでは防げず**、SBOM・依存固定・最小権限・署名検証・TPRM・インシデント訓練を組み合わせる必要がある

「**自社のセキュリティを完璧にしても、ベンダーが落ちたら同時に落ちる**」という現実が、サプライチェーン攻撃の本質です。
