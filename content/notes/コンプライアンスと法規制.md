# コンプライアンスと法規制

> [!summary]
> マネタイズを目指すなら避けて通れない。GDPR（EU個人データ）、日本の個人情報保護法、PCI DSS（決済情報）、SOC 2（B2B SaaS）。各々の「最低限ここは守る」基準を知って、サービス設計の早い段階で対応コストを下げる。

## なぜサービス設計の早期に必要か

「あとで対応する」がほぼ機能しない領域。GDPRやAPPIは**データ収集の設計時点で**「目的」「最小化」「保管期間」が決まっていないとあとから直すのに大手術。SOC 2は監査対象期間（通常6か月〜1年）を遡って証拠が必要。

## GDPR（EU 一般データ保護規則）

### 概要

- 2018年5月適用開始
- EU内の個人のデータを扱う**全ての事業者**が対象（EU外でもサービス提供してれば該当）
- 違反時の罰金：年間売上の **4%** または **€2000万** のうち高い方

### 個人開発者でも押さえるポイント

- **法的根拠（lawful basis）** — 同意 / 契約履行 / 法的義務 / 正当な利益 等の6つから1つ
- **データ最小化** — 必要なデータだけ収集
- **目的制限** — 収集時に説明した目的以外で使わない
- **保管期間** — 必要期間を超えて保持しない
- **データ主体の権利** — アクセス・訂正・削除（**忘れられる権利**）・ポータビリティ
- **侵害通知** — 個人データ侵害から **72時間以内**に監督機関に通知

### 設計に効くこと

- ユーザーアカウント削除機能（**ハードデリート**できる構造に）
- データエクスポート機能（JSON出力等）
- 同意管理（Cookieバナーだけでなくメール配信等の同意も）
- DPA（Data Processing Agreement）を SaaS / 下請けと締結

## 個人情報保護法（日本、APPI）

### 概要

- 2003年制定、2022年に大幅改正
- 改正で「個人関連情報」「仮名加工情報」「越境移転規制」が強化
- 違反時の罰金：法人は **最大1億円**

### 重要な定義

- **個人情報** — 特定の個人を識別できる情報
- **個人データ** — DB等に体系的に整理された個人情報
- **要配慮個人情報** — 病歴・犯罪歴・人種等、明示同意が必要
- **個人関連情報** — Cookie等。第三者提供時に本人同意要件あり

### 設計に効くこと

- **利用目的の特定と通知**（プライバシーポリシーで具体的に）
- **越境移転** — 海外サーバ（Vercel us-east-1等）に保存する場合、ユーザーへ国名等の情報提供が必要
- **漏えい時の報告** — 個人情報保護委員会への報告と本人通知（72時間以内目安、改正で義務化）
- **保有個人データの開示請求** への対応窓口

### GDPRとの違い

- GDPRは「全EU居住者」で広い。APPIは日本居住者対象だが、日本企業に対する効力が強い
- 「忘れられる権利」は両方あるが要件が異なる
- DPAに相当する委託契約はAPPIにも要件あり

## PCI DSS（カード決済情報）

### 適用範囲

カード会員データ（PAN、有効期限、サービスコード、CVV、PIN）を**保存・処理・伝送する**事業者が対象。最新は **v4.0**（2024年4月から、2025年3月以降必須）。

### 個人開発者の現実解

決済情報を**自分で保持しない**設計にする：

- **Stripe / Square / GMOペイメント** などのPaymentサービスを使う → カード情報は彼らが扱い、自分はトークンしか持たない
- **Apple Pay / Google Pay** で `pan` 自体が手元に来ない
- 自社では **`cus_xxx`** のような顧客IDだけ持つ

これで **PCI DSS の SAQ A**（最も軽い自己評価）に該当。膨大な要件の大半が免除される。

直接 PAN を扱う設計は、フルPCI DSS要件（暗号化、ネットワーク分離、12のメイン要件）が来るので個人では避けるべき。

## SOC 2（B2B SaaSの信頼通貨）

### 概要

- 米国公認会計士協会（AICPA）が定義
- 5つの **Trust Service Criteria** から選んで監査される
  - **Security**（必須）
  - Availability
  - Processing Integrity
  - Confidentiality
  - Privacy
- **Type 1**：ある時点での設計の妥当性
- **Type 2**：一定期間（6か月〜1年）の運用実績

### B2B SaaSなら必要になるタイミング

- 中規模以上の企業に売り込む時、ベンダーセキュリティアセスメントで要求される
- 契約書のセキュリティ条項に「SOC 2 Type 2 を取得していること」が入ってくる

### 取得への準備

- ポリシー文書整備（情報セキュリティポリシー、IR手順、変更管理 等）
- アクセス管理・MFA徹底
- ログ・監査の記録（[[セキュリティロギング設計]]）
- 変更管理（PR review、CI/CD）
- インシデント対応プロセス（[[インシデントレスポンス]]）
- ベンダー管理（使うSaaSの審査記録）

実際の監査は外部監査法人を使う。規模が出てから着手する領域だが、**創業初期から「SOC 2 で問われる項目」を意識した運用**にしておくと、後の取得コストが激減する。

## ISO 27001（参考）

国際標準の情報セキュリティマネジメントシステム（ISMS）。SOC 2 と並ぶ「セキュリティ品質の証明書」。日本企業にはこちらの方が馴染みが深い。

## 個人開発から事業化への準備チェックリスト

- [ ] プライバシーポリシーを書いた（GDPR・APPI 対応）
- [ ] 利用規約を書いた
- [ ] Cookie同意（必要に応じて）
- [ ] アカウント削除機能（ハードデリート）を実装
- [ ] データエクスポート機能を実装
- [ ] 決済はStripe等のPSP経由で PCI DSS SAQ A に
- [ ] インシデント時の72時間通知体制（連絡先・手順）
- [ ] ログ・監査の記録を残す（SOC 2を意識）
