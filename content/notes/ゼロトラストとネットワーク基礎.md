# ゼロトラストとネットワーク基礎

> [!summary]
> 「内側 vs 外側」という古典的境界防御から「**全ての接続を検証する**」ゼロトラストへの転換。TLS設定、HSTS、DNS security（DNSSEC、DoH/DoT）、証明書管理。前回の [[トンネルの分類と定義]] とも繋がる領域。

## ゼロトラストとは

NIST SP 800-207 の定義：「**暗黙の信頼を排除し、すべてのアクセスを継続的に検証する**」。

従来の「VPNの中は信頼、外は信頼しない」モデルが破綻した理由：

- リモートワークで境界が曖昧化
- SaaS時代に「内側」が分散
- 内部に侵入されたときにラテラルムーブメントを止められない
- BYOD・契約者など多様な接続元

### コアの原則（NIST 7原則）

1. すべてのデータソース・サービスはリソース
2. すべての通信は安全（場所に依存せず）
3. **接続ごとに**アクセスを許可（セッション単位）
4. **ポリシーで動的に**判断（コンテキスト：ユーザー、デバイス、時刻、振る舞い）
5. すべての資産の整合性を継続的に監視
6. 認証・認可は**接続前に**毎回
7. データを集めてセキュリティ姿勢を改善

### 実装パターン

- **ZTNA**（Zero Trust Network Access）— SaaS的VPN代替（Cloudflare Access、Tailscale、Zscaler、Twingate）
- **BeyondCorp**（Google発祥）— デバイス + ユーザー認証ベース
- **SDP**（Software-Defined Perimeter）

[[Tailscale]] は WireGuard ベースのZTNA実装で、個人開発でも実用範囲。

## TLSの実戦設定

### 最低ライン

- **TLS 1.3** をデフォルト、**1.2 が最低**、それ以下は無効
- 強い cipher suite のみ（Mozilla SSL Configuration Generator が便利）
- 証明書は **Let's Encrypt + ACME** で自動更新
- **OCSP Stapling** で revocation チェックの遅延回避
- 検証：[Mozilla Observatory](https://observatory.mozilla.org/)、[SSL Labs](https://www.ssllabs.com/ssltest/)

### HSTS（HTTP Strict Transport Security）

ブラウザに「次回からHTTPSしか使うな」を覚えさせる：

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age` は1年以上推奨
- `preload` を指定して https://hstspreload.org/ に登録すれば、初回アクセスからHTTPS強制
- 本番ドメインで設定したら**戻すのが難しい**（ブラウザのキャッシュが切れるまで）ので、staging で確認してから

### 証明書ピンニング（Certificate Pinning）

特定の証明書（or 中間CA）以外を信用しない強化策。Webでは HPKP が廃止された経緯あり、**モバイル/IoT** で使うのが現実的。誤設定で自分のアプリを永久ブリックする危険性がある。

### mTLS

クライアント側も証明書で認証。**サーバ間通信** や Zero Trust ネットワーク内で使われる。

## DNS security

### DNSSEC

DNS応答に署名を付けて改ざん検知。レゾルバが検証。導入率はまだ部分的だが、TLDレベル（`.jp`等）では普及。

### DoH（DNS over HTTPS）/ DoT（DNS over TLS）

DNSクエリをHTTPS / TLSで暗号化。盗聴・改ざんを防ぐ。Cloudflare 1.1.1.1 や Google 8.8.8.8 が提供。

ブラウザ側ではFirefox / Chrome が DoH対応。

### DNS の落とし穴

- DNSキャッシュポイズニング
- ドメイン乗っ取り（レジストラのアカウント侵害）
- **Subdomain Takeover**（[[Web脆弱性の実装防御]]）— dangling CNAME
- DNS amplification攻撃（DDoS）

## ネットワーク基礎の押さえどころ

### 暗号化のレイヤ別対応

| レイヤ | プロトコル例 | 暗号化 |
|---|---|---|
| L7（アプリ） | HTTP/2、HTTP/3、gRPC | TLS（基本必須） |
| L4（トランスポート） | TCP、UDP、QUIC | TLS / DTLS |
| L3（ネットワーク） | IP | IPsec |
| L2（リンク） | Ethernet | MACsec |

### VPNの分類

[[トンネルの分類と定義]] を参照。L3 VPN（WireGuard、IPsec、OpenVPN）と L4 / アプリ層（SSH、HTTP CONNECT、WebSocket）で守備範囲が異なる。

### ファイアウォール／WAF

- **L3/L4 ファイアウォール** — IP/ポート単位のフィルタ
- **WAF（Web Application Firewall）** — L7、HTTPリクエストの内容検査。**AWS WAF、Cloudflare、Cloudflare Pages のWAF機能**
- **rate limit / bot mitigation** がWAFと一体化していることが多い

### CDNと組み合わせる効果

- DDoSの吸収
- TLS終端をCDN側で
- WAFをCDN側で
- 攻撃元IPのカスケード遮断

Vercelは内部的にCloudflareやAWS CloudFrontを使っている。

## ゼロトラストを個人プロジェクトに

完全なZTAは過剰だが、**部分的な原則の適用**は有効：

1. SSHでのpassword認証を完全廃止 → 鍵 + パスフレーズ + MFA
2. インフラ管理画面（AWS Console等）はMFA必須
3. 開発者の本番アクセスは ZTNA 経由（Cloudflare Access、Tailscaleなど）
4. プライベートサービスは **Tailnetでのみ公開**
5. 「IP制限」を信用しすぎない（強い保護にはならない）
