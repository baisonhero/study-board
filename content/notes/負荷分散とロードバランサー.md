# 負荷分散とロードバランサー

> [!summary]
> 複数サーバに**トラフィックを分散**して可用性とスケーラビリティを得る仕組み。L4とL7で性質が大きく違う。AWSのALB/NLB/CLB、ヘルスチェック、SSL終端、スティッキーセッション。

## なぜ負荷分散が必要か

- **可用性**：1台死んでもサービス継続
- **スケーラビリティ**：負荷に応じて台数増減
- **メンテナンス**：1台ずつ更新できる
- **柔軟性**：B/Gデプロイ、カナリアリリース

## L4 vs L7 ロードバランサー

| | L4 (Transport) | L7 (Application) |
|---|---|---|
| 判断基準 | IP + ポート | HTTPメソッド、URLパス、ヘッダ、Cookie |
| プロトコル | TCP/UDP汎用 | HTTP/HTTPS（基本） |
| パフォーマンス | 速い | やや重い（パケット解析あり） |
| 機能 | シンプル | 多機能（パスルーティング、書き換え等） |
| 例 | NLB、HAProxy(L4) | ALB、Nginx、HAProxy(L7) |

### L7にしかできないこと

- パスベースルーティング（`/api/*` は API、`/static/*` は CDN へ）
- ホスト名ベース（`api.example.com` と `web.example.com` で振り分け）
- ヘッダ書き換え、認証
- 圧縮、レスポンスキャッシュ
- WAF統合

## 負荷分散アルゴリズム

| アルゴリズム | 動き | 適する場面 |
|---|---|---|
| **ラウンドロビン** | 順番に振る | サーバ性能均等、ステートレス |
| **重み付きラウンドロビン** | 重み比率で振る | サーバ性能不均等 |
| **最少接続数（Least Connections）** | 接続数が少ないサーバへ | 処理時間まちまち |
| **最少応答時間** | RTT短いサーバへ | レスポンス重視 |
| **IPハッシュ** | 送信元IPで振り先決定 | スティッキー必要時 |
| **ランダム** | ランダム | 簡易 |
| **URL/Cookieハッシュ（L7）** | URL/Cookie値でハッシュ | キャッシュヒット率重視 |

## ヘルスチェック

LB配下のサーバが「生きてるか」を定期確認：

- **エンドポイント**：`/health`、`/healthz` など軽量応答
- **間隔**：5〜30秒が一般的
- **失敗閾値**：N回連続失敗で除外
- **回復閾値**：M回連続成功で復帰

> [!tip] 良いヘルスチェックの設計
> 「DBに繋がるか」「依存サービスに繋がるか」も含めるか？ → 含めると依存サービス障害時に**全台 unhealthy** になって連鎖障害。**自分自身の生存だけ**にして、依存先障害は別アラートで検知する設計が安全。

## TLS 終端（SSL Termination）

LBで TLS を終端してバックエンドにはHTTPで投げる方式が一般的：

```
Client --[HTTPS]--> [LB（証明書）] --[HTTP]--> [App Server]
```

メリット：
- 証明書管理が1箇所に集中
- バックエンドサーバの負荷軽減
- L7ルーティングのために中身を見る必要

セキュリティ：
- **VPC内** のLB→Appは原則安全とされるが、ゼロトラスト的には mTLS 等で再暗号化推奨
- **Pass-through**（LBを素通り）にする選択肢もある（NLB + Nitro Enclave 等）

## スティッキーセッション

「同じユーザーは同じサーバへ」を保証。Cookie や IPベース：

- Cookie ベース：LBが `AWSALB` のような Cookie を発行
- IPベース：送信元IPでハッシュ

**現代的にはステートレス設計**（セッション情報を Redis 等の共有ストアに）が推奨。スティッキーは負荷分散効率を下げる。

## AWS のロードバランサー

| サービス | レイヤ | 特徴 |
|---|---|---|
| **ALB** (Application Load Balancer) | L7 | HTTP/HTTPS、WebSocket、HTTP/2、パスルーティング、認証統合（Cognito） |
| **NLB** (Network Load Balancer) | L4 | TCP/UDP、超高性能、静的IP、ソースIP保持 |
| **CLB** (Classic Load Balancer) | L4/L7 | レガシー、新規利用非推奨 |
| **GLB** (Gateway Load Balancer) | L3 | サードパーティアプライアンス用、IPS等 |

### ALB vs NLB の選択

- **ALB**: HTTPなら基本これ。WAF、認証、リッチなルーティング
- **NLB**: 非HTTP、高性能要件、固定IPが必要、TCPパススルー

### Target Group

LB配下のターゲット群（EC2、ECS Task、Lambda、IP）。ヘルスチェックはこの単位で。

## CDN との関係

CDN（CloudFront、Cloudflare、Fastly）は**世界に分散したエッジロケーション**で：

- 静的コンテンツのキャッシュ配信
- TLS終端をエッジで
- DDoS吸収
- WAF配置

階層イメージ：
```
[Client] → [CDN edge] → [LB] → [App]
```

CDNでL7的な振る舞いも可能（Cloudflare Workers、CloudFront Functions）。

## クラスタ内のLB（Service Mesh寄り）

Kubernetesでは：
- **Service** が L4 LB（kube-proxy が iptables/IPVS で実装）
- **Ingress Controller**（Nginx、Traefik、Istio）が L7
- **Service Mesh**（Istio、Linkerd）でPod間mTLS、リトライ、サーキットブレイカ

## トラブルシューティング

```bash
# LBへのHTTP接続確認
curl -v https://lb.example.com/

# 個別ターゲットへの直接接続
curl -v http://10.0.10.5:8080/health

# ヘルスチェック状態（AWS）
aws elbv2 describe-target-health --target-group-arn <arn>

# LBのログ（ALB Access Log）
aws s3 cp s3://my-alb-logs/AWSLogs/.../...log.gz - | gunzip | head
```

ありがちな問題：

- **502 Bad Gateway**：バックエンドが落ちてる、ヘルスチェックNG
- **504 Gateway Timeout**：バックエンドの応答遅い、idle timeout超過
- **「ヘルスチェックは通るのに本番はNG」**：ヘルスチェックが軽すぎ（DB接続入ってない等）
