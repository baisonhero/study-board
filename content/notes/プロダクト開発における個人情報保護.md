# プロダクト開発における個人情報保護

> [!summary]
> マネタイズ可能な個人サービスを作る上で**避けて通れない領域**。日本の個人情報保護法（APPI）、EUのGDPR、それぞれの定義・要件、Privacy by Design、実装パターン、よくある落とし穴。サービス設計の早期に手を打てば対応コストは激減する。[[コンプライアンスと法規制]] が法律全般の概観なら、こちらは**実装ガイド寄り**。

## なぜ最重要トピックか

個人情報を扱うプロダクトでこれをミスると：

- **罰金**：日本（個人情報保護法、最大1億円）、EU（GDPR、年間売上の4% or €2000万）、加州（CCPA/CPRA、件数×$2,500〜$7,500）
- **信頼喪失**：一度漏らすと顧客は戻ってこない
- **サービス停止命令**：監督官庁の権限
- **訴訟リスク**：集団訴訟、損害賠償

「あとで対応」が**ほぼ機能しない**領域でもある。データ収集の設計時点で「目的」「最小化」「保管期間」が決まっていないと、後から直すのに大手術になる。

## 1. 個人情報保護法（日本、APPI）

### 法律の構造

正式名称：**個人情報の保護に関する法律**（2003年制定、2017年・2020年・**2022年大幅改正**、3年ごと見直し）。

監督機関は **個人情報保護委員会（PPC）**。

### 重要な「データ分類」

法律上の用語は5階層あり、それぞれ規制が違う：

| 用語 | 定義 | 取り扱い規制 |
|---|---|---|
| **個人情報** | 特定の個人を識別できる情報（氏名、生年月日との組合せ、個人識別符号） | 利用目的の特定・通知 |
| **個人データ** | 個人情報DBを構成する個人情報（≒整理された個人情報） | + 安全管理措置、第三者提供制限 |
| **保有個人データ** | 事業者が開示・訂正・削除権限を持つ個人データ | + 開示請求への対応 |
| **要配慮個人情報** | 人種・信条・社会的身分・病歴・犯罪歴・被害事実等 | **明示的な同意が必須**、第三者提供のオプトアウトNG |
| **個人関連情報** | Cookie、IPアドレス等、それ単体では個人特定できない情報 | 第三者提供時、提供先で個人データになる場合は本人同意要件 |
| **仮名加工情報** | 他の情報と照合しないと特定できないように加工 | 内部利用に緩和（2020年改正で新設） |
| **匿名加工情報** | 復元不可能なまでに加工 | 第三者提供OK |

> [!info] 「個人関連情報」の重要性（2022年改正）
> Cookie・広告ID・閲覧履歴は単体では個人情報ではないが、**提供先で他のデータと紐づけて個人を特定できる場合**は同意要件が発生する。広告タグやリターゲティングを実装する時に直接効く規制。

### 事業者の義務（要点）

1. **利用目的の特定** — できるだけ具体的に（「事業活動に用いる」では不十分）
2. **目的の通知・公表** — 取得時にプライバシーポリシー等で示す
3. **目的外利用の禁止** — 同意なしには変更できない
4. **適正な取得** — 不正な手段で取らない
5. **安全管理措置** — 4類型で実施
6. **従業員監督・委託先監督** — 委託先が漏らしても委託元の責任
7. **第三者提供制限** — 原則同意要、例外あり
8. **本人からの開示・訂正・削除請求への対応** — 原則無料、2週間以内目安
9. **漏えい時の報告・通知** — PPCへの報告と本人への通知（2022年改正で義務化）

### 安全管理措置の4類型

| 類型 | 例 |
|---|---|
| **組織的** | 責任者、従業員教育、点検、内部規程 |
| **人的** | 雇用契約での秘密保持、教育、誓約書 |
| **物理的** | 施錠、入退館管理、機器盗難防止 |
| **技術的** | アクセス制御、暗号化、ログ監視、セキュリティソフト |

クラウドサービスを使う場合、技術的安全管理措置の多くは**事業者側の機能**に依存する（=委託先監督が重要）。

### 越境移転規制（2022年改正で強化）

外国にある第三者に個人データを提供する場合：

1. **本人の同意**
2. または **基準適合国**（EU、英国）への移転
3. または **委託先が基準を満たす体制**を確保

> [!warning] Vercel/Supabaseでの注意点
> Vercel（米国）、Supabase（プロジェクトのregion指定可、東京もある）は**外国にある第三者**にあたる。プライバシーポリシーで以下を明示する：
> - 移転先国名（米国、シンガポール 等）
> - 当該国の制度の概要
> - 移転先が講じている保護措置

### 漏えい時の対応

2022年改正で**義務化**：

- **PPCへの報告**：速報（**3〜5日以内目安**）+ 確報（**30日以内、不正アクセス起因なら60日以内**）
- **本人への通知**：必要に応じて速やかに

報告対象は「漏えいの蓋然性」レベルで判断（実害確定を待たない）。

## 2. GDPR（EU 一般データ保護規則）

### 適用範囲（重要）

EU内に**サービスを提供している**事業者は対象（拠点がEU外でも）。日本のサービスでもEUユーザーが使えれば該当する。

### 個人データの定義

「**識別された、または識別可能な自然人に関する全ての情報**」。日本法より広い。IPアドレス・Cookie・デバイスIDも明確に個人データ扱い。

### 法的根拠（lawful basis）— 6つから1つ

データ処理には法的根拠が必須：

1. **同意**（consent）
2. **契約履行**（contract）
3. **法的義務**（legal obligation）
4. **重要利益**（vital interests）
5. **公的任務**（public task）
6. **正当な利益**（legitimate interests）

サービス開発で多用するのは「同意」と「契約履行」と「正当な利益」。各処理ごとに根拠を決め、プライバシーポリシーで開示。

### データ主体の権利（拒否されたら必ず対応）

| 権利 | 内容 |
|---|---|
| **アクセス権** | 自分のデータを見たい |
| **訂正権** | 間違いを直して |
| **削除権**（忘れられる権利）| 全部消して |
| **処理制限権** | 一時的に止めて |
| **データポータビリティ** | 機械可読形式で持ち出したい |
| **異議申立権** | 「正当な利益」根拠の処理に異議 |
| **自動意思決定の対象とならない権利** | プロファイリング拒否 |

これらの権利を**実装で対応**する必要がある（=削除APIやエクスポート機能）。

### 重要要件

- **DPO（Data Protection Officer）**：大規模処理や要配慮データ処理は専任者必須
- **DPIA（Data Protection Impact Assessment）**：高リスク処理の事前評価
- **侵害通知**：監督機関に **72時間以内**、影響大なら本人にも

## 3. その他の主要法域（参考）

### 米国

- **CCPA / CPRA**（カリフォルニア）— 加州民の権利保護、年間売上等の閾値あり
- **HIPAA**（医療情報）— 医療サービス特化
- 連邦レベルの包括法はまだない（**州ごとパッチワーク**で複雑）

### 中国

- **個人情報保護法（PIPL）**（2021年）— **域外適用、データローカライゼーション要件**、罰金最大年間売上の5%。中国ユーザー向けサービスは要注意

### その他アジア

- **PDPA**（シンガポール、タイ、マレーシア）— 各国独自
- **韓国 PIPA** — 厳格な部類

### 業界別

- **PCI DSS**（決済情報、[[コンプライアンスと法規制]]）
- **HIPAA**（米国医療）
- **FERPA**（米国教育）

## 4. Privacy by Design（PbD）— 7原則

設計段階からプライバシーを組み込む思想。Ann Cavoukian提唱、GDPR第25条で法的義務化。

1. **事後対応ではなく事前予防** — 漏えい後の対応ではなく、起きる前に対策
2. **デフォルトでプライバシー保護** — 何もしなくても保護される設定が初期値
3. **設計に組み込む** — 後付け機能ではない
4. **ゼロサムではなくフルファンクショナル** — プライバシーと機能性は両立可能
5. **ライフサイクル全体で保護** — 取得・処理・保管・廃棄まで
6. **可視性と透明性** — 何をしているかユーザーに見える
7. **ユーザーのプライバシーを尊重** — ユーザー中心

## 5. 実装パターン

### データ最小化（Data Minimization）

```typescript
// ❌ 雑に全部取る
type User = {
  id: string
  name: string
  email: string
  birthdate: string  // 本当に必要？
  phone: string      // 本当に必要？
  address: string    // 本当に必要？
}

// ✅ 機能ごとに最小化
type AuthUser = { id: string; email: string }
type ProfileUser = { id: string; displayName: string }
type ShippingUser = { id: string; address: string }  // 配送機能ONの時だけ
```

「**取らない**」のが最強の保護。

### 削除機能（ハードデリート）の実装

```sql
-- ❌ ソフトデリート（削除フラグ）だけだと「忘れられる権利」を満たさない
UPDATE users SET deleted_at = NOW() WHERE id = $1;

-- ✅ 関連データもまとめて物理削除
BEGIN;
DELETE FROM orders WHERE user_id = $1;
DELETE FROM sessions WHERE user_id = $1;
DELETE FROM users WHERE id = $1;
-- ログテーブルはハッシュ化IDに置換するか、anonymizeする
UPDATE audit_logs SET actor_id = 'deleted-user' WHERE actor_id = $1;
COMMIT;
```

例外：法令で保管が必要な期間（税法7年、特商法3年等）は削除できない場合がある。**「削除を依頼されたが法令により保管が必要なデータ」を別管理**できる設計に。

### エクスポート機能（データポータビリティ）

ユーザーが自分のデータをJSON等で取得できるエンドポイント：

```
GET /api/me/export
→ 200 OK
Content-Type: application/json
Content-Disposition: attachment; filename="my-data-2026-04-26.json"

{
  "profile": {...},
  "orders": [...],
  "preferences": {...}
}
```

GDPR Article 20の要件。日本法では明文義務はないが、**「保有個人データの開示請求」**への現実的な対応手段になる。

### 同意管理

複数の処理に対して**処理ごとに**同意を管理：

```typescript
type Consents = {
  termsOfService: { agreed: boolean; version: string; timestamp: Date }
  marketingEmail: { agreed: boolean; timestamp: Date }
  analytics: { agreed: boolean; timestamp: Date }
  thirdPartyShare: { agreed: boolean; timestamp: Date }
}
```

- **「全部同意」ボタンの罠** — 個別同意を強制する法域がある（GDPRのspecific consent要件）
- **撤回も簡単に** — 同意取得と同等の容易さで撤回可能でなければならない（GDPR Article 7(3)）

### Cookie同意（バナー）

EU向けには**事前同意**（opt-in）が原則：

- 必須Cookie（セッション、CSRF）— 同意不要
- 機能Cookie（言語設定）— 通知ベースで可
- 分析Cookie（GA等）— 同意必須
- 広告Cookie — 同意必須

実装はカスタムでも、**Cookiebot、OneTrust** 等のCMP（Consent Management Platform）を使うのが楽。

### ログからのPII除外

[[セキュリティロギング設計]] と直結：

- ロガーに **redaction（マスキング）パターン**を設定
- メールアドレス、電話番号、クレジットカード番号等は自動的に `***@example.com` `****-****-****-1234` に
- 例外スタックトレースに含まれるリクエストボディも要マスキング

```typescript
// pino logger例
const logger = pino({
  redact: {
    paths: ['req.body.password', 'req.body.email', '*.creditCard'],
    censor: '[REDACTED]',
  },
})
```

### 仮名化（Pseudonymization）

「IDは保つがすぐには個人が特定できない」状態にする：

```typescript
// 内部分析テーブルに渡す前に仮名化
function pseudonymize(userId: string): string {
  return crypto
    .createHmac('sha256', process.env.PSEUDONYM_SECRET)
    .update(userId)
    .digest('hex')
}
```

GDPRでは仮名化が推奨技術として明示。**「鍵」を別管理**することで、必要時には逆引き可能、漏えい時は影響を限定。

### 匿名化（Anonymization）

「**復元不可能**なまでに加工」する。集計値、ノイズ付加（差分プライバシー）、k-匿名性。**完全な匿名化は技術的に難しい**ことが知られているので、研究用途以外では仮名化を選ぶことが多い。

## 6. プライバシーポリシーに必ず書くこと

### 日本法の必須記載事項

- 事業者の名称・住所・代表者
- 利用目的（具体的に）
- 取得方法
- 第三者提供の有無と範囲
- 委託先の有無
- **越境移転がある場合**：移転先国名、当該国の制度概要、講じている保護措置
- 開示等請求の窓口
- 苦情申し出の窓口
- Cookie等の利用（個人関連情報の取扱）

### GDPR追加要件

- 法的根拠（lawful basis）を処理ごとに
- データ主体の権利の説明
- 保管期間
- DPOの連絡先（該当する場合）

### よくあるミス

- **「事業活動に必要な範囲」**などの抽象的な目的記載 → 具体化必須
- **更新時の通知漏れ** → 重要な変更は再同意取得
- **テンプレート流用** → 実態と乖離したポリシーは違反

## 7. 越境移転対応（個人開発で重要）

### Vercel / Supabase / AWS のリージョン選び

| サービス | 日本リージョン | 越境移転 |
|---|---|---|
| **Vercel** | 一部Edge機能あり、デプロイ地域は限定的 | 米国移転になることが多い |
| **Supabase** | `ap-northeast-1`（東京）選択可 | プロジェクト作成時に選ぶ |
| **AWS** | `ap-northeast-1`、`ap-northeast-3` | 自分で制御 |
| **Cloudflare** | エッジは日本含む全世界 | データ保存先による |

設計時の判断ポイント：

1. 個人データは**日本リージョン**で保存
2. ログ・分析データも可能な限り日本
3. 越境せざるを得ない部分は明示してプライバシーポリシーに

### DPA（Data Processing Agreement）

GDPR Article 28で要求。SaaSベンダーと結ぶ「処理委託契約」。

- AWS DPA、Vercel DPA、Supabase DPA は**標準で公開済み**、利用規約に含まれていることが多い
- 確認して保管しておく
- 監査時の証拠になる

## 8. よくある落とし穴

### コードレベル

- **ローカルストレージにJWTを保存** → XSSで奪取可能、PIIが含まれていれば漏えいに直結
- **エラーログにリクエストボディ全部** → パスワード・PIIが流出
- **デバッグエンドポイントを本番に放置** → `/admin/debug` で全ユーザー情報露出
- **GitHub Issueに本物のメールアドレスを貼る** → スクレイピングで収集される
- **不要なフィールドを返すAPI** → React Server Components のpropsに混入

### 運用レベル

- **テストデータに本番データを使う** → 開発環境からの漏えい
- **削除依頼を「無効フラグ」だけで対応** → 法令違反
- **同意取得時のUX dark pattern** → 「拒否」ボタンを小さく見えにくく → GDPR違反
- **プライバシーポリシーを更新したのに通知なし** → 同意取り直しが必要なケース
- **退職者のアクセス権が残っている** → 委託先監督義務違反

### 法律理解レベル

- **「Cookieは個人情報じゃないから」** → 個人関連情報として規制対象
- **「日本法だから外国は関係ない」** → EUユーザーが使えれば GDPR、加州ユーザーがいれば CCPA
- **「無料サービスだから関係ない」** → 法律は対価の有無を問わない
- **「B2Bだから個人情報じゃない」** → 法人の従業員情報は個人情報に該当することが多い

## 9. 設計時チェックリスト

新サービス・新機能を作る時に確認：

- [ ] このデータは**本当に必要**か？（最小化）
- [ ] **法的根拠**は何か？（同意/契約/正当利益）
- [ ] 同意を取るならその**取り方**は適切か？
- [ ] **保管期間**は決まっているか？（無期限はNG）
- [ ] **削除機能**は実装されているか？
- [ ] **エクスポート機能**は提供できるか？
- [ ] **ログにPII**は混入しないか？
- [ ] 利用する**SaaSの所在地**は適切か？（越境移転）
- [ ] **DPA**は結んだか／確認したか？
- [ ] **プライバシーポリシー**を更新したか？
- [ ] **越境移転の記載**は反映したか？
- [ ] 委託先の**安全管理措置**は確認したか？
- [ ] **漏えい時の対応**手順は決まっているか？

## 10. 個人マネタイズサービスの最低ライン

> [!tip] 創業期の現実解
> 完璧を目指して動けないより、最低ラインを早く満たして動く方が良い。

1. プライバシーポリシーと利用規約を**自分の言葉で**書く（テンプレ丸コピは避ける）
2. SupabaseのリージョンはAP-Northeast-1（東京）
3. Vercel/SupabaseのDPAを確認・保存
4. アカウント削除機能を実装（ハードデリート）
5. データエクスポート機能を実装
6. ログ・エラーレポートのPII除外設定
7. Cookie同意（必要に応じて、EU対応するなら）
8. 漏えい時の連絡先（PPC、ユーザー通知方法）を手元に
9. 委託先（SendGridでメール送るなら、Stripeで決済するなら）の所在地把握
10. アクセスログを最低1年保管
