# セキュリティロギング設計

> [!summary]
> 「事故が起きた後でも追える」ように設計するロギング。何を残し何を残さないか（PII除外）、構造化、改ざん防止、保管期間。学習中の [[OpenTelemetry]] とも直結する領域で、observabilityの基盤がそのままIRの基盤になる。

## 目的の多層性

ログが必要な理由は複数あって、混同するとどれも中途半端になる：

| 目的 | 何のため | 性質 |
|---|---|---|
| **デバッグログ** | 開発者が問題を再現する | 詳細、短命 |
| **アプリログ** | 通常運用の挙動把握 | バランス |
| **監査ログ（audit log）** | 「誰が何をしたか」の記録 | **改ざん防止が必須** |
| **セキュリティログ** | インシデント検知・追跡 | 改ざん防止、長期保管 |

[[インシデントレスポンス]] では特に下2つが重要。

## 監査ログに必ず含める情報

- **Who**: アクター（ユーザーID、サービスアカウント）
- **What**: 何の操作（操作種別、対象リソースID）
- **When**: タイムスタンプ（ISO 8601、UTC、ミリ秒精度以上）
- **Where**: ソース（IP、リージョン、デバイス情報）
- **Why/How**: 認証経路（API key / session / token、MFA有無）
- **Result**: 成功・失敗（失敗時は理由カテゴリ）

OWASP Logging Cheat Sheet が良い参考。

## 絶対にログに入れてはいけない

- **パスワード**、API key、secret、JWT、session token そのもの
- クレジットカード番号、CVV、SSN等
- 暗号化前の機密データ
- メールアドレスや電話番号などの個人特定情報（**ハッシュ化された識別子**にする）
- 健康情報、政治信条、宗教等の特別カテゴリ（GDPR Article 9）

> [!warning] エラーメッセージにも注意
> 例外を catch して `error.toString()` を素朴にログ出力すると、SQLクエリにパスワードが混入していたり、HTTPボディが丸ごと出てしまうことがある。**シリアライザに redaction（マスキング）**を入れる設計が安全。

## 構造化ログ

JSON 1行＝1イベントが基本。利点：

- 機械可読でクエリしやすい（CloudWatch Logs Insights、Loki、Datadog）
- フィールド単位でインデックス・集計ができる
- スキーマを揃えればチーム横断で見える

最低限のフィールド：

```json
{
  "timestamp": "2026-04-26T10:30:00.123Z",
  "level": "info",
  "service": "api",
  "actor_id": "u_a1b2c3",
  "action": "order.create",
  "resource": "order/12345",
  "trace_id": "8f3e2d1c...",
  "span_id": "4b5a6c7d",
  "result": "success"
}
```

## OpenTelemetryとの関係

[[OpenTelemetry]] のログシグナルを使うと、トレース（traceId）と同じコンテキストで紐づく：

- 1リクエストに紐づく全ログを traceId で串刺し検索
- アプリログ → ロガー → OTel Collector → Backend（Tempo/Loki/Grafana Cloud）

セキュリティ用途のログも、別チャンネルではなくOTelに統合した方が運用が楽。

## 改ざん防止

監査ログは「**書き込んだら誰も書き換えられない**」状態を作る：

- **Append-only ストア**：S3 + Object Lock、CloudWatch Logs（編集不可）、専用アカウントへの転送
- **暗号学的チェーン**：前のログのハッシュを次に含めて改ざんを連鎖検知（Merkle tree、blockchainの応用）
- **WORM（Write Once Read Many）**：S3 Glacier Compliance、Azure immutable blob

個人プロジェクトでも **本番ログは別AWSアカウントのS3にミラー**、で「攻撃者がログ消して証拠隠滅」を最低限防げる。

## 保管期間

| ログ種別 | 推奨保管期間 |
|---|---|
| デバッグ | 7〜30日 |
| アプリログ | 30〜90日 |
| 監査ログ | **最低1年、推奨3〜7年** |
| セキュリティログ | 監査と同等以上 |
| GDPR要件のログ | 3年程度（事業による） |
| PCI DSS | 1年（過去3か月は即時アクセス可） |

長期保管は安価層（S3 Glacier等）に階層化。

## アラート設計（検知の入口）

ログを集めるだけでは見ない。「**異常パターン**」を事前定義してアラート：

- 認証失敗の急増（ブルートフォース）
- 管理API への普段ない地理からのアクセス
- 1ユーザーの異常な数のリクエスト
- `iam:*` `kms:Disable*` `s3:DeleteBucket` などの破壊的API
- 課金額の異常スパイク

EventBridge + SNS / Slack webhook で自分のスマホに飛ばす。

## ログのコスト管理

セキュリティ志向で全てを記録するとコストが爆発する。バランス策：

- **粒度を絞る**：通常ルートは INFO、重要操作だけ DEBUG
- **サンプリング**：高頻度イベントはサンプリング（[[OpenTelemetry]] のtail samplingが応用できる）
- **階層化保存**：直近 = 高速ストレージ、過去 = Glacier 等
- **ログ削減のためにメトリクスに変換**（カウントだけで十分なものはCounter化）
