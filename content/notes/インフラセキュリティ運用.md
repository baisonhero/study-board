# インフラセキュリティ運用

> [!summary]
> [[セキュリティ学習ロードマップ]] の運用編。**ハードニング**（OS・ミドルウェア要塞化）、**パッチ管理**、**脆弱性管理**、**CIS Benchmarks** の実装。アプリ層のAppSecとは別レイヤで、サーバ運用者の責任範囲。

## ハードニングとは

「**OS・ミドルウェアを攻撃に強くする**」初期設定 + 定期点検。

### よくあるOSハードニング項目

#### Linux

- **不要サービスの停止・削除**
- **SSH強化**（[[SSH実運用]]）：パスワード認証無効、root禁止、特定ユーザーのみ
- **firewall**：default deny、必要ポートのみ
- **SELinux / AppArmor**：強制有効
- **ファイル権限**：`/etc/shadow` 600、`/tmp` のsticky bit
- **kernel パラメータ**：`net.ipv4.tcp_syncookies=1`、`net.ipv4.conf.all.rp_filter=1`
- **auditd**：監査ログ取得
- **fail2ban**：ブルートフォース対策
- **自動アップデート**：`unattended-upgrades`（Ubuntu）

#### Windows

- 既定の管理者アカウント無効化
- BitLocker でディスク暗号化
- Windows Defender / SmartScreen
- グループポリシーで一括統制

#### コンテナ

- non-root実行
- read-only filesystem
- capabilities drop
- distroless / alpine ベースイメージ
- 詳細は [[コンテナとKubernetesセキュリティ]]

## CIS Benchmarks（具体的な設定推奨）

[[セキュリティ標準とフレームワーク]] でも触れた、**製品ごとの具体的な設定推奨値集**。

主要なBenchmark：

- CIS AWS Foundations Benchmark
- CIS Azure Foundations Benchmark
- CIS Linux Benchmarks（RHEL、Ubuntu等）
- CIS Kubernetes Benchmark
- CIS Docker Benchmark
- CIS Microsoft Windows Benchmarks
- CIS Cisco Benchmarks

各 Benchmark に **Level 1**（基本）と **Level 2**（高セキュリティ）。

### 自動チェックツール

- **CIS-CAT**：CIS純正、有料（CIS SecureSuite Member）
- **OpenSCAP**：OSS、SCAP規格でCIS含む各種Benchmark対応
- **kube-bench**：Kubernetes向け（OSS）
- **AWS Security Hub**：AWS Benchmark を自動評価

## パッチ管理

### 重要な原則

- **定期適用**：月次・週次のリズム
- **緊急パッチ**：CVE Critical は数日以内
- **テスト環境で検証**：dev → staging → prod
- **ロールバック計画**：失敗時の戻し方を事前準備
- **ベースライン管理**：「パッチレベル X 以上を全インスタンスに」

### Linuxの自動アップデート

#### Ubuntu/Debian

```bash
sudo apt install unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades
# /etc/apt/apt.conf.d/50unattended-upgrades で設定

# セキュリティのみ自動の場合
"Unattended-Upgrade::Allowed-Origins" {
    "${distro_id}:${distro_codename}-security";
};
```

#### RHEL/CentOS

```bash
sudo dnf install dnf-automatic
sudo systemctl enable --now dnf-automatic.timer
# /etc/dnf/automatic.conf で設定
```

### コンテナのパッチ

「コンテナはパッチしない、再ビルド」が原則：

- ベースイメージ更新
- イメージリビルド + デプロイ
- 古いイメージは積極的に削除

### クラウドの自動パッチ

- **AWS Systems Manager Patch Manager** — EC2 / オンプレに patch baseline 適用
- **AWS Inspector** — 脆弱性スキャン
- **Azure Update Management** — Azure Arc でハイブリッド対応
- **GCP OS Patch Management** — Compute Engineに

## 脆弱性管理

### CVE と CVSS

- **CVE**（Common Vulnerabilities and Exposures）：脆弱性ID
- **CVSS**（Common Vulnerability Scoring System）：深刻度スコア（[[セキュリティ標準とフレームワーク]]）

新しいCVEが出るたび、自社環境への影響を判断するワークフロー：

```
CVE公開 → 影響評価（自社利用バージョンか？）→ 優先度決定 → パッチ計画 → 適用 → 確認
```

### 脆弱性スキャナ

| ツール | 対象 |
|---|---|
| **Nessus** | サーバ / ネットワーク総合（商用） |
| **OpenVAS** | OSS、Nessusの開発元の昔のfork |
| **Qualys** | クラウド型、エンタープライズ |
| **AWS Inspector** | EC2 / ECR / Lambda |
| **Trivy** | コンテナイメージ、IaC（OSS） |
| **Grype** | コンテナイメージ（OSS） |

### SBOM 連動

[[SBOM]] を成果物として持っておくと、新CVE 公開時に「影響範囲を機械的に検索」できる。

## 監査ログとSIEM

### 集めるべきログ

| 種別 | 内容 |
|---|---|
| **OSログ** | `/var/log/auth.log`、`secure`、syslog |
| **アプリログ** | アクセスログ、エラーログ |
| **ネットワーク** | VPC Flow Logs、firewall ログ |
| **クラウド監査** | CloudTrail、Cloud Audit Logs、Activity Log |
| **DB監査** | PostgreSQL audit、Aurora audit |
| **コンテナ** | Kubernetes audit log、container stdout |

詳しくは [[セキュリティロギング設計]]。

### SIEM（Security Information and Event Management）

ログを集約・相関して脅威を検知：

- **AWS Security Hub** — クラウド寄り、findings集約
- **Splunk** — エンタープライズの王道、高価
- **Elastic Stack（旧ELK）** — OSS主体
- **Datadog Cloud SIEM** — APMと統合
- **Microsoft Sentinel** — Azure ネイティブ
- **Chronicle**（Google）— 大規模ログ向け

### SOC（Security Operations Center）

24/365でセキュリティ監視・対応する組織。MSSP（Managed SSP）にアウトソースするケース多し。

## 鍵・証明書管理

### TLS証明書

- **Let's Encrypt** + Certbot で無料・自動更新
- AWS Certificate Manager（ACM）— AWS内部利用なら無料
- 期限切れ警告は事前に（30日前）
- 全社ドメインを一覧化

### SSH鍵 / クラウドのアクセスキー

[[シークレット管理]] と [[SSH実運用]] 参照。長期キー削減と rotation 自動化。

### 暗号鍵管理（KMS / HSM）

- AWS KMS、Azure Key Vault、GCP Cloud KMS
- 重要キーの **rotation** と **アクセス監査**
- HSMが必要な要件（FIPS 140-2 Level 3 等）は専用 HSM サービス

## バックアップとDR

### バックアップ戦略 3-2-1

- **3つのコピー**
- **2種類のメディア**
- **1つはオフサイト**

クラウド時代の意訳：
- 本番ストレージ + バックアップ + 別アカウント or 別リージョン

### RPO / RTO

- **RPO**（Recovery Point Objective）：許容できるデータ損失時間
- **RTO**（Recovery Time Objective）：許容できる復旧時間

要件に応じて：

| RPO/RTO | 戦略 |
|---|---|
| 数時間でOK | スナップショット + 別リージョンレプリケーション |
| 数分 | リアルタイム複製、warm standby |
| ~ゼロ | active-active、multi-region |

### ランサムウェア対策

- バックアップへの **書き込み権限を分離**（バックアップ用 IAM ロール）
- **Object Lock**（S3 / Azure Blob）で改ざん防止
- 一定期間 immutable に
- バックアップの**復旧テスト**を定期実施

## 構成変更管理

### Change Management

- 本番への変更は**チケット化**＋承認
- IaC（[[IaCとTerraform基礎]]）で変更履歴を保持
- リリースウィンドウとフリーズ期間
- ロールバック計画
- 事後レビュー（変更が成功したか、想定外がないか）

## DevSecOps（[[アプリケーションセキュリティ ツール分類]]）統合

CI/CDパイプラインにセキュリティチェック組み込み：

```
コード push
  ↓
Secret Scan（gitleaks）
  ↓
SAST（Semgrep / CodeQL）
  ↓
SCA（Trivy / Snyk）
  ↓
ビルド + イメージスキャン（Trivy）
  ↓
署名（Cosign）
  ↓
IaC scan（Checkov / tfsec）
  ↓
deploy
  ↓
DAST（OWASP ZAP）on staging
  ↓
prod deploy
```

各ステップでブロック条件を決めて、人間判断ゲートを最小に。
