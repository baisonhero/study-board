# ソフトウェアサプライチェーン強化

> [!summary]
> [[SBOM]] で「何が入っているか」を可視化したら、次は「**それが本物か・どう作られたか**」を担保する段階。Sigstore で署名検証、SLSA で来歴保証レベル、lockfile integrity、dependency confusion 対策。

## 全体像

サプライチェーン攻撃の入口は大きく4つ：

1. 依存ライブラリ自体が汚染される（typosquatting、メンテナ乗っ取り）
2. ビルドパイプラインが侵害される（CI/CDの改ざん）
3. レジストリ偽装（dependency confusion）
4. 配布段階での改ざん（CDN、ミラーサーバ）

それぞれに対策層がある。

## Sigstore — 署名で本物を証明する

### 何の問題を解くか

「このバイナリは本当にメンテナが作ったものか？」を**鍵管理なしで**検証可能にする。OSS開発者が長期鍵を保持・管理するのは現実的でないので、**Keyless Signing** という新発想で解決した。

### 構成要素

- **cosign** — 署名・検証CLI（containers / blobs / OCI artifacts対応）
- **Fulcio** — 短命証明書を発行する認証局（OIDC IDから発行）
- **Rekor** — 透明性ログ（全署名を公開ログに記録、改ざん不能）

### 使い方の流れ

```bash
# OIDCでログイン → 短命証明書発行 → 署名 → Rekorに記録
cosign sign-blob --bundle sig.bundle ./binary

# 検証
cosign verify-blob --bundle sig.bundle --certificate-identity=... ./binary
```

GitHub Actions から Sigstore に署名する仕組みは GitHub OIDC Provider 経由で簡単に組める。

## SLSA — 来歴の保証レベル

**SLSA**（Supply-chain Levels for Software Artifacts、サルサと読む）は「このアーティファクトの作られ方をどこまで信頼できるか」を**4段階**で定義したフレームワーク。

| レベル | 概要 |
|---|---|
| **Build L1** | provenance（来歴）が文書化されている |
| **Build L2** | ホストされたビルドサービスで作られ、署名済みのprovenance |
| **Build L3** | 改ざん耐性のあるビルドプラットフォーム、隔離されたビルド環境 |
| **Build L4** | （現在は廃止、将来的にPolicy領域へ統合） |

### 提供する証拠（provenance）

- 何のソースから（commit hash）
- どのビルダーが（GitHub Actions のどのworkflowなど）
- どのコマンドで
- 何のアーティファクトを作ったか

GitHub Actionsには `slsa-github-generator` があって、L3を比較的簡単に達成できる。

## Lockfile Integrity

`package-lock.json` `pnpm-lock.yaml` `Cargo.lock` `go.sum` 等の役割：

- **完全に同じ依存ツリー**を再現する
- **インストール時にハッシュ検証** — 公開後に改ざんされても検出
- CIで `npm ci` や `pnpm install --frozen-lockfile` を使って lockfile からの逸脱を防ぐ

**コミットしないのは事故のもと**。lockfileは必ずGitに含める。

## Dependency Confusion（混同攻撃）

2021年に判明した攻撃。社内プライベートパッケージと同名のパッケージを公開レジストリに上げると、設定によっては公開版が優先されて配信される。Microsoft、Apple、Tesla等が被害を受けた。

防御：

- パッケージマネージャを **socket-style scope** で運用（`@yourcompany/pkg-name`）
- レジストリ優先度の明示設定（`.npmrc` の `registry` と scope-specific registry）
- 内部レジストリのみで公開しない名前空間予約
- npm の `npm config set fund false` ではなく実装的には `publishConfig.access` を private にする

## Typosquatting

人気パッケージの綴り違いを公開して、タイポしたユーザーに悪意あるコードを実行させる。`reqeusts`（本物は`requests`）、`crossenv`（本物は`cross-env`）等。

防御：

- pre-commit にパッケージ追加時のレビュー手順
- Snyk、Socket.dev のようなパッケージ評価サービスでスコア確認
- 新しい依存追加は社内承認フロー

## CIパイプラインの保護

- **Pinned actions** — `actions/checkout@v4` ではなく `actions/checkout@<full-sha>` で固定（タグは書き換え可能）
- **Permissions最小化** — workflow毎に `permissions:` を `read-all` ではなく必要最低限
- **Secrets分離** — 信頼できるmainブランチからしか触れないシークレット
- **OIDC with Cloud** — long-lived AWS access keyを置かず、GitHub OIDC経由で短命credential取得

## 個人プロジェクトでの最低ライン

1. lockfile を必ずコミット、CIで `--frozen-lockfile` 系を使う
2. **Dependabot を有効化**（CLAUDE.mdのDevSecOpsルールにも記載）
3. GitHub Actions の SHA pinning（または `dependabot.yml` で自動更新）
4. リリース成果物に `cosign sign-blob` を CI から実行
5. 主要な依存は Snyk Advisor / Socket.dev でスコアチェック
