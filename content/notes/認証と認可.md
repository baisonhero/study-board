# 認証と認可（AuthN/AuthZ）

> [!summary]
> 認証 = 「誰か」を確かめる、認可 = 「何ができるか」を決める。Webアプリの脆弱性のうち最も影響が大きいのが認可不備（[[OWASP Top 10]] A01 Broken Access Control）。SupabaseアプリではRLS（Row Level Security）が認可の中核。

## 用語

- **AuthN（Authentication / 認証）**：あなたは誰？を確かめる
- **AuthZ（Authorization / 認可）**：あなたは何ができる？を決める

両方を区別しないと議論がブレる。例えばOAuthは認可フレームワーク、OIDCはOAuthを土台にした認証層。

## 押さえるべきプロトコル・パターン

### OAuth 2.0

リソースアクセスの認可フレームワーク。Authorization Code flow（PKCE付き）が現在のベスト。Implicit flowは非推奨。Client Credentials flowはサーバ間通信用。

学習ポイント：
- リダイレクトURIの完全一致検証の重要性
- PKCEがなぜSPA / モバイルで必須か
- リフレッシュトークンの管理（rotation、revoke）

### OpenID Connect（OIDC）

OAuth 2.0の上に乗った**認証層**。ID Token（JWT）でユーザーアイデンティティを表現。Auth0、Cognito、Supabase Authはこれの実装。

### JWT（JSON Web Token）の落とし穴

- `alg: none` 受け入れ脆弱性（古いライブラリ）
- 署名検証を忘れて payload だけ読む実装ミス
- 公開鍵と秘密鍵の取り違え（HS256 ↔ RS256 confusion）
- 有効期限を長くしすぎる
- ペイロードに機密情報を入れる（base64エンコードなだけで暗号化されてない）

### セッション管理

cookie属性の正しい設定：
- `Secure` — HTTPS限定
- `HttpOnly` — JSからアクセス不可（XSS対策）
- `SameSite=Lax` または `Strict`（CSRF対策、Lax がデフォルト推奨）
- `Path` / `Domain` を最小限に

セッション固定攻撃対策：ログイン時にセッションIDを再発行する。

## 認可モデル

| モデル | 概要 | 適する場面 |
|---|---|---|
| **RBAC**（Role-Based Access Control）| ロールに権限を紐づけ、ユーザーをロールに割り当てる | 中小規模、ロール数が静的 |
| **ABAC**（Attribute-Based Access Control）| 属性（ユーザー属性 × リソース属性 × 環境）でルール評価 | 細かい制御、動的条件 |
| **ReBAC**（Relationship-Based Access Control）| 関係グラフで判定（GoogleのZanzibar論文） | コラボツール、SNS的なもの |

[[Supabase RLS]] は実質ABACのSQL版。ユーザーIDとレコードのowner_idなど属性比較でアクセス制御する。

## Supabaseの場合の最重要トピック：RLS

- すべてのテーブルで **`ENABLE ROW LEVEL SECURITY`** をデフォルトでオン
- ポリシー未設定 = アクセス不可（fail-secure）
- `auth.uid()` でログインユーザーIDを取得
- INSERT/UPDATE/DELETE/SELECT 別にポリシーを書く
- service_role キーはRLSをバイパスする → サーバ側コードでも極力使わない

> [!warning] Supabaseの典型的事故パターン
> RLSを無効のままテーブルを作って公開してしまう。`anon` キーは公開前提なので、RLSがないと全データが見える。本番デプロイ前にRLS確認を必ずチェックリストに入れる。

## MFA（多要素認証）

実装パターン：
- TOTP（Google Authenticator等）— 標準的
- WebAuthn / Passkey — 最も安全、UXも良い、推奨方向
- SMS — 非推奨（SIMスワップ攻撃）
- メール — フォールバック以上の意味は薄い

## OWASP Top 10との対応

- **A01 Broken Access Control** — 認可ミス全般
- **A07 Identification and Authentication Failures** — 認証実装ミス、弱いパスワード許容、ブルートフォース対策不足
