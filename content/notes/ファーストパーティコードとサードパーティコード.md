# ファーストパーティコードとサードパーティコード

> [!summary]
> 「自分で書いたコード」と「他人が書いて使わせてもらってるコード」の区別。AppSecツールの守備範囲（[[SAST]] vs [[SCA]]）はこの区別に対応する。信頼境界の設計や責任分界を考えるときの基本概念。

## 定義

| 用語 | 意味 |
|---|---|
| **ファーストパーティコード**（first-party code）| 自分（or自社）が書いたコード。ソースコードリポジトリにコミットしているもの |
| **サードパーティコード**（third-party code）| 他人が書いたものを使わせてもらっているコード。OSSライブラリ、SDK、フレームワーク等 |
| **セカンドパーティコード**（second-party code）| 比較的レアな表現。取引先や提携先など「第三者ではないが完全に自分でもない」コード。実務ではあまり使われない |

## なぜこの区別が重要か

### セキュリティ：守備範囲が違う

[[アプリケーションセキュリティ ツール分類]] で出てきたツールは、この境界に沿って分かれている：

| 対象 | ツールカテゴリ | 何を見るか |
|---|---|---|
| ファーストパーティ | [[SAST]] | 自分が書いたコードのバグ・脆弱なパターン |
| サードパーティ | [[SCA]] | 使ってる依存ライブラリの既知CVE・ライセンス |

### 責任分界

- **ファーストパーティの脆弱性** → 自分が直す
- **サードパーティの脆弱性** → 上流にレポート、それまでは依存を更新するか回避策を入れる

### サプライチェーン攻撃の文脈

「自分のコードが綺麗でも、依存先で悪意のあるコードが混入したら終わり」というのが [[サプライチェーン攻撃]] の本質。Log4Shell も xz-utils backdoor もサードパーティ起点。だから [[SBOM]] や [[SCA]] が重要視される。

### 信頼境界

セキュリティ設計では「**信頼できるコード境界**」を引く必要がある。サードパーティは原則として「**完全には信頼しない**」のが基本姿勢：

- 依存最小化（minimal dependencies）
- バージョン固定（lockfile）
- 改ざん検出（hash verification、[[SBOM]]）
- 自動アップデート + テスト（[[Dependabot]] + CI）

## 「コードスニペット」との混同に注意

**ファーストパーティコード ≠ コードスニペット**。

- **ファーストパーティコード** = 出どころの話（誰が書いたか）
- **コードスニペット** = 粒度の話（短い切れ端）

自分が書いたソースコード**全体**を指したいなら「ソースコード」「コードベース」「ファーストパーティコード」が正解。「コードスニペット」と呼ぶと「短いサンプル例」のニュアンスになるので意味がズレる。
