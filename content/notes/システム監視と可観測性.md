# システム監視と可観測性

> [!summary]
> 「**動いてるか」だけ見るMonitoring**から、「**なぜ動いていないか**」まで分かる**Observability**へ。Three Pillars（メトリクス・ログ・トレース）、SLI/SLO/SLA、Prometheus + Grafana、CloudWatch、Datadog。学習中の[[OpenTelemetry]]ともど直結。

## Monitoring vs Observability

| 観点 | Monitoring（伝統） | Observability（現代） |
|---|---|---|
| 思想 | 既知の問題を検知 | 未知の問題も追える状態 |
| 単位 | システム | リクエスト |
| 主な情報 | メトリクスとアラート | メトリクス＋ログ＋トレース |
| 質問 | 「いま壊れている？」 | 「なぜ遅い？どこで？」 |

Observabilityは**システムの内部状態を外から推測できる性質**。Monitoringは**手段**の一つ。

## Three Pillars（3本柱）

### 1. Metrics（メトリクス）

**数値の時系列**。CPU使用率、リクエスト数、レスポンスタイム等。

| 種類 | 例 |
|---|---|
| **Counter** | リクエスト総数（単調増加） |
| **Gauge** | 現在のCPU使用率（増減） |
| **Histogram** | レスポンスタイムの分布 |
| **Summary** | パーセンタイル付きHistogram |

主なツール：Prometheus、CloudWatch、Datadog Metrics

### 2. Logs（ログ）

**離散イベントの記録**。詳細な情報が含まれるが、量が多い。

主なツール：CloudWatch Logs、Loki、Elasticsearch、Datadog Logs

[[セキュリティロギング設計]] と直結。

### 3. Traces（トレース）

**1リクエストが複数サービスを通る経路**を追跡。マイクロサービス時代に必須。

- **Trace** = リクエスト全体
- **Span** = 個別の処理単位
- **TraceID / SpanID** で串刺し

主なツール：[[OpenTelemetry]] + Jaeger / Tempo / X-Ray / Datadog APM / SigNoz

## OpenTelemetry（OTel）

CNCF プロジェクト。**ベンダー非依存**でメトリクス・ログ・トレースを取得・送信する標準。

```
[App] --[OTel SDK]--> [OTel Collector] --> [Backend（Tempo / Datadog / X-Ray）]
```

学習中のため詳細は別途、`~/Documents/devs/study/otel/` の手元実習を参照。

## SLI / SLO / SLA / SLB

### SLI（Service Level Indicator）

「**実際に測ってる指標**」。例：

- 可用性：成功リクエスト数 / 全リクエスト数
- レイテンシ：99%ileのレスポンスタイム
- スループット：req/sec

### SLO（Service Level Objective）

「**SLIの目標値**」。例：

- 月次の availability SLI が 99.9% 以上
- 週次の p99 latency が 500ms 以下

### SLA（Service Level Agreement）

「**SLOの契約版**」、外部公開する保証。違反時にcredit / refund。

### SLB（Service Level Budget = Error Budget）

SLO 100% から実績を引いたもの。「**まだ壊していい余裕**」がどれだけあるか：

- 月次SLO 99.9% → エラー予算 0.1% = 月43.2分
- これを使い切ったらリリース凍結 → 復旧に注力

エラー予算の運用は SREの肝。

## RED と USE と Four Golden Signals

監視で何を見るか、定番のフレーム3つ：

### RED（リクエスト中心、サービス向け）

- **Rate**：req/sec
- **Errors**：エラー率
- **Duration**：レイテンシ

### USE（リソース中心、サーバ向け）

- **Utilization**：使用率
- **Saturation**：飽和度
- **Errors**：エラー数

### Four Golden Signals（Google SRE Book）

- Latency（遅延）
- Traffic（流量）
- Errors（エラー）
- Saturation（飽和）

REDとUSEを組み合わせた感じ。**まずこの4つを取れば大半カバー**。

## アラート設計

### 良いアラートの条件

- **症状ベース**（症状を述べる）：「APIエラー率10%超」
- **原因推測アラート ❌**：「CPU90%超」（CPUが高いのは結果、症状はSLO違反）
- **対応可能（Actionable）**：受け取った人が次の手が打てる
- **緊急性に応じた段階**：critical（即対応）/ warning（観察）/ info（記録）

### アラート疲れの回避

- 大量のアラート → 麻痺 → 重要なものを見逃す
- 「**ノイズと shadowed なアラートを削る**」を継続的に
- 「**alert fatigue**」を避けるためのレビュー会

### オンコール

- 24/365 の対応体制
- ローテーション
- ランブック整備（[[インシデントレスポンス]]）
- 連絡経路ツール：PagerDuty、Opsgenie、VictorOps

## 主要ツール

### Prometheus + Grafana（OSS主流）

```
[App]      --[expose /metrics]--> [Prometheus] --> [Grafana]
                                       ↓
                                  [Alertmanager] -> Slack/PagerDuty
```

- **Prometheus**：時系列DB + スクレイピング型
- **Grafana**：可視化（ダッシュボード）
- **Alertmanager**：アラート集約・通知
- **Loki**：Prometheus互換のログDB（Grafana Labs製）
- **Tempo**：トレースDB

PrometheusはPull型（Prometheusがアプリに `/metrics` を取りに行く）。スケール時は **Thanos** や **Cortex** で水平展開。

### AWS CloudWatch

- メトリクス（標準＋カスタム）
- Logs（取り込み量で課金、コスト注意）
- Synthetics（外形監視）
- ServiceLens（X-Ray と統合してトレース可視化）
- Container Insights（ECS / EKS）

統合は深いがUIや検索はやや弱い。

### Datadog

- APM（トレース）+ Metrics + Logs + Synthetics + RUM が一体
- 高品質UI、検索強い
- 高い（フルセット利用すると月数万円〜）

### New Relic / Dynatrace / Splunk Observability

商用APMの主要プレイヤー。エンタープライズ向け。

### SigNoz / OpenSearch / Elastic Stack

OSS or 自前ホスト派の選択肢。OTelとの相性良し。

## ダッシュボード設計

### 階層化（USE Method の派生）

1. **エグゼクティブ層**：SLO達成率、ユーザー影響
2. **サービス層**：RED metrics、エラー上位
3. **インフラ層**：USE metrics、ノード状態

### ダッシュボードのベストプラクティス

- 1ダッシュボードは1テーマ（あれもこれも詰めない）
- 時系列は **複数のサービスを並べて見られる**ように
- p50/p90/p99 の3つを並べて表示
- **アラート閾値**を線で書き込む
- スクリーンに常時表示できるTV ダッシュボードを別途作る

## クラウド固有の観測

### AWS

- **CloudWatch Metrics**：基本
- **CloudWatch Logs Insights**：クエリ言語でログ集計
- **X-Ray**：トレース
- **CloudWatch Container Insights / Lambda Insights**：コンテナ・サーバレス
- **EventBridge**：イベント駆動の通知
- **VPC Flow Logs**：ネットワークフロー
- **Application Signals**（新しい、SLO定義機能内蔵）

### Kubernetes

- **kube-state-metrics**：K8sリソース状態をメトリクス化
- **node-exporter**：ノードメトリクス
- **Prometheus Operator** + **kube-prometheus-stack**（Helm chart）
- **kubectl top** で簡易確認

## Sleeper Agent / Synthetic Monitoring

- 定期的に「外側からアプリを叩く」監視
- Synthetics、Pingdom、UptimeRobot
- 5分おきにヘルスチェック → 失敗で通知
- 個人プロジェクトでも UptimeRobot 無料枠で十分

## ログとの関係（再掲）

メトリクスは**集計後**、ログは**詳細**。両方必要：

- 高頻度・サマリ → Metrics
- 低頻度・詳細・調査用 → Logs

[[セキュリティロギング設計]] と組み合わせて運用。
